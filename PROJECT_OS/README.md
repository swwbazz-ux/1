# Операционная система проекта корпоративного портала ООО «Коппер Рисорсез»

## Назначение

`PROJECT_OS` — минимальная долговременная память проекта. Она нужна для:

- передачи контекста между чатами;
- синхронизации ChatGPT и Codex;
- восстановления состояния проекта без чтения всей истории переписки.

## Роли

### Пользователь

- владелец продукта;
- принимает окончательные решения;
- утверждает визуальный и функциональный результат.

### ChatGPT

- продуктовый консультант;
- UX- и визуальный куратор;
- технический контролёр;
- формирует задания Codex;
- проверяет commits и результаты через GitHub.

### Codex

- исполнитель-разработчик;
- изучает репозиторий;
- меняет только разрешённые файлы;
- выполняет проверки;
- делает commit и push;
- обновляет `PROJECT_OS/CHATGPT_SYNC.md`.

### GitHub

- долговременный и проверяемый источник состояния проекта.

## Рабочий процесс

1. Пользователь и ChatGPT обсуждают задачу.
2. ChatGPT готовит точное задание.
3. Пользователь передаёт задание Codex.
4. Codex выполняет задачу.
5. Codex проводит проверки.
6. Codex обновляет `PROJECT_OS/CHATGPT_SYNC.md`.
7. Codex делает commit и push.
8. Пользователь сообщает ChatGPT SHA commit.
9. ChatGPT самостоятельно проверяет commit через GitHub.
10. Пользователь принимает результат или отправляет на доработку.

## Правила памяти

- Важные решения нельзя оставлять только в чате.
- Система должна развиваться постепенно.
- Новые документы добавляются только при реальной необходимости.
- `PROJECT_OS` не заменяет документы `71_` и `72_`, а служит точкой входа к ним.
- Рабочая ветка берётся из `PROJECT_OS/CHATGPT_SYNC.md`, а фактический HEAD всегда дополнительно проверяется через GitHub.
- SHA в журнале может обозначать последний содержательный commit; для фиксации SHA служебного commit не создаётся рекурсивный commit.
- [PROJECT_ROADMAP.md](PROJECT_ROADMAP.md) управляет новым продуктовым и визуальным треком публичного сайта.
- Старый публичный дизайн не является основой нового сайта.
- Только пользователь закрывает продуктовые этапы.

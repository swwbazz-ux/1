# Сценарий пилотного запуска

Дата создания: 19.06.2026

## Назначение

Документ описывает короткий порядок первой пилотной проверки производственного MVP.

Цель: пройти систему по ролям, создать или проверить производственные события, открыть отчеты, выгрузить Excel и понять, какие доработки нужны перед реальным запуском.

## Формат пилота

Первый пилот можно провести в двух режимах.

### Режим 1. Демонстрационный пилот

Используется подготовленная демо-база.

Подходит для:

- показа руководству;
- проверки логики по ролям;
- первичной проверки отчетов;
- обучения ключевых участников перед реальной сменой.

### Режим 2. Пилот на реальной смене

Используется реальная смена или ее часть.

Подходит для:

- проверки, смогут ли сотрудники вводить данные в реальном темпе;
- сверки отчетов с текущими Excel-формами;
- выявления неудобных действий и недостающих полей.

## Подготовка перед пилотом

### 1. Обновить код и базу

Проверить, что используется актуальная ветка:

```text
codex/mechanic-downtime-dashboard
```

Проверить проект:

```text
python manage.py check
python manage.py test users
```

### 2. Подготовить демо-данные

Для демонстрационного пилота выполнить:

```text
python manage.py seed_demo_scenario
```

Команда готовит:

- роли;
- демо-доступы;
- сотрудников;
- смены;
- назначения;
- активный рейс;
- выполненный рейс;
- шаблон отчета;
- открытый и закрытый простой.

### 3. Открыть карту интерфейсов

Адрес:

```text
/interfaces/
```

Через карту удобно открыть все основные экраны.

## Демо-коды доступа

| Роль | Код | Что проверяет |
|---|---:|---|
| Администратор | 1000 | справочники, доступы, админка |
| Водитель самосвала | 2000 | смена, назначение, активный рейс, завершение рейса |
| Машинист экскаватора | 3000 | создание рейса |
| Горный мастер | 4000 | назначение самосвала под экскаватор |
| Диспетчер | 5000 | пульт контроля, отчеты, Excel |
| Руководство | 6000 | управленческая витрина |
| Механик | 7000 | простои техники |

## Порядок пилотной проверки

### Шаг 1. Проверка входа и карты интерфейсов

Действия:

1. Открыть `/interfaces/`.
2. Проверить, что видны основные интерфейсы и отчеты.
3. По очереди войти кодами 5000 и 6000.

Ожидаемый результат:

- система открывает интерфейс по роли;
- диспетчер попадает на пульт;
- руководство попадает на витрину.

### Шаг 2. Проверка расстановки

Роль:

- горный мастер, код 4000.

Действия:

1. Открыть экран горного мастера.
2. Выбрать экскаватор.
3. Выбрать самосвал.
4. Создать назначение.

Ожидаемый результат:

- назначение появляется в системе;
- водитель видит назначение;
- диспетчер видит назначение на пульте.

### Шаг 3. Проверка водителя

Роль:

- водитель, код 2000.

Действия:

1. Открыть смену, если смена не открыта.
2. Проверить стартовые показатели техники.
3. Подтвердить назначение кнопкой `Принял`.
4. Проверить активный рейс.
5. Завершить рейс кнопкой `Выполнено`.

Ожидаемый результат:

- водитель выполняет минимум действий;
- назначение становится принятым;
- завершенный рейс попадает в отчет.

### Шаг 4. Проверка машиниста экскаватора

Роль:

- машинист экскаватора, код 3000.

Действия:

1. Открыть экран машиниста.
2. Выбрать назначенный самосвал.
3. Выбрать породу или груз.
4. Выбрать точку разгрузки.
5. Создать рейс.

Ожидаемый результат:

- рейс создается;
- объем считается из правил кубатуры;
- тоннаж считается по плотности породы, если она заполнена;
- водитель видит активный рейс;
- диспетчер видит активный рейс.

### Шаг 5. Проверка диспетчерского пульта

Роль:

- диспетчер, код 5000.

Действия:

1. Открыть `/dispatcher/control/`.
2. Проверить активные рейсы.
3. Проверить назначения без подтверждения.
4. Проверить принятые назначения.
5. Проверить незакрытые смены.
6. Проверить открытые простои.
7. Проверить журнал последних диспетчерских действий.

Ожидаемый результат:

- диспетчер видит текущую смену в процессе;
- активные проблемы не теряются;
- есть быстрые переходы к отчетам.

### Шаг 6. Проверка простоев

Роль:

- механик, код 7000.

Действия:

1. Открыть экран механика.
2. Проверить открытые простои.
3. Создать простой по технике, если нужен новый пример.
4. Закрыть простой.
5. Открыть отчет `/reports/downtimes/`.
6. Проверить сводку ОР ККД/СКДР.
7. Выгрузить Excel.

Ожидаемый результат:

- простои видны как события, а не просто текст;
- отчет показывает длительность, статус, технику и причину;
- Excel содержит лист `ОР ККД СКДР`.

### Шаг 7. Проверка отчета по объемам

Роль:

- диспетчер, код 5000.

Действия:

1. Открыть `/reports/volume/`.
2. Проверить таблицу рейсов.
3. Проверить фильтры.
4. Проверить группировку по часу: `/reports/volume/?group_by=completed_hour`.
5. Выгрузить Excel.

Ожидаемый результат:

- рейсы, объемы и тоннаж отображаются;
- переходящие рейсы помечаются отдельно;
- группировка по часу работает;
- Excel соответствует выбранным фильтрам и группировке.

### Шаг 8. Проверка суточного отчета заказчику

Роль:

- диспетчер, код 5000.

Действия:

1. Открыть `/reports/customer-daily/`.
2. Проверить сменные блоки.
3. Проверить план, факт, отклонение.
4. Проверить итоги с начала месяца.
5. Проверить блок сверки со старой Excel-формой.
6. Выгрузить Excel.

Ожидаемый результат:

- отчет открывается без ручной сборки;
- Excel содержит рабочий суточный отчет;
- Excel содержит лист сверки со старой формой.

### Шаг 9. Проверка витрины руководства

Роль:

- руководство, код 6000.

Действия:

1. Открыть `/reports/management/`.
2. Проверить суточные KPI.
3. Проверить выполнение плана.
4. Проверить день против ночи.
5. Проверить динамику за 7 дней.
6. Выгрузить Excel.

Ожидаемый результат:

- руководство видит картину без ручного отчета;
- есть суточная и недельная динамика;
- Excel-выгрузка работает.

### Шаг 10. Проверка чеклиста пилота

Роль:

- диспетчер, администратор или руководство.

Действия:

1. Открыть `/reports/pilot-checklist/`.
2. Проверить матрицу покрытия Excel-форм.
3. Проверить ссылки на рабочие экраны.
4. Проверить прогресс этапа.

Ожидаемый результат:

- видно, какие формы уже закрыты MVP;
- видно, какие формы перенесены после пилота;
- понятно, что именно сверять с диспетчерской.

## Что фиксировать во время пилота

Фиксировать отдельно:

- какие действия пользователи не понимают;
- где нужно меньше кликов;
- какие поля не хватает;
- какие поля лишние;
- какие данные пользователи не смогут вводить в карьере стабильно;
- где отчет расходится со старой Excel-формой;
- какие Excel-формы нужны ежедневно;
- какие Excel-формы можно оставить после MVP;
- какие показатели нужны руководству в первую очередь.

## Минимальный результат пилота

Пилот считается технически успешным, если:

- все роли входят в систему;
- рейс создается и завершается;
- объем и тоннаж считаются;
- диспетчер видит активные события;
- отчеты открываются;
- Excel-выгрузки формируются;
- витрина руководства открывается;
- расхождения зафиксированы списком, а не остаются в устных замечаниях.

## Что делать после пилота

После пилота нужно создать список доработок по группам:

1. Критично до реального запуска.
2. Желательно до реального запуска.
3. Можно после запуска.
4. Отдельный большой модуль.

Отдельно выделить:

- интерфейс водителя;
- интерфейс машиниста экскаватора;
- интерфейс горного мастера;
- диспетчерский пульт;
- отчеты заказчику;
- витрину руководства;
- механический модуль;
- справочники и админку.

## Вывод

Первый пилот должен дать не идеальную промышленную систему, а проверенный список фактов:

- базовый производственный цикл работает или нет;
- какие данные реально можно получать из карьера;
- какие отчеты уже можно заменить;
- какие расхождения нужно закрыть перед запуском;
- какие большие модули идут следующей очередью.

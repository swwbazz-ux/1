# Концепция интерфейса Горного диспетчера

Дата создания: 23.06.2026.

## Дополнение 24.06.2026: адаптивная ширина гаража самосвалов

Для рабочего экрана Горного диспетчера принято правило по гаражу самосвалов:

- слоты самосвалов в гараже должны совпадать по размеру со слотами гаража экскаваторов;
- гараж самосвалов строится колонками по 12 слотов: 52 свободных самосвала - 4 колонки со скроллбаром, 48 - 4 колонки без скроллбара, 36 - 3 колонки, 24 - 2 колонки, 12 - 1 колонка, 0 - гараж скрывается;
- пустые парковочные места дорисовываются только как бледные силуэты самосвалов, без номеров, подписей, статусов и статусбаров;
- горизонтального скролла в гараже быть не должно;
- если вертикальный скроллбар нужен, место под него резервируется внутри расчетной ширины гаража и фактически забирается у активной зоны смены;
- когда вертикальный скроллбар больше не нужен, эта ширина возвращается активной зоне смены.

## Дополнение 24.06.2026: подписи состояния в гараже экскаваторов

В гараже экскаваторов принято повторить полезный паттерн из гаража самосвалов: реальная техника в слоте показывает короткую текстовую подпись состояния внизу плитки.

Правило отображения:

- у реальных экскаваторов показывается подпись состояния: `работает`, `простой`, `ожидание`, `ремонт` и аналогичные короткие статусы;
- подпись ставится в нижнюю часть слота и не должна перекрывать PNG-иконку техники;
- пустые парковочные места остаются без текста, номера, статуса и статусбара;
- занятые слоты экскаваторов, которые временно заменены бледным силуэтом парковочного места, также не показывают подпись.

## Дополнение 24.06.2026: нумерация самосвалов в гараже

В гараже самосвалов видимая подпись техники должна показывать только бортовой номер без префикса `TR-`.

Правило нумерации для демо-сетки Горного диспетчера:

- первый самосвал начинается с номера `10`;
- номер `53` не используется в гараже самосвалов, потому что это эвакуатор;
- для 52 самосвалов демо-ряд отображается как `10...52, 54...62`;
- все DnD-чипы и подписи, которые берут имя из плитки самосвала, используют эту же видимую нумерацию.

## Дополнение 24.06.2026: цвет простоя техники

Для Горного диспетчера закреплено статусное правило: `простой` не является желтым ожиданием, а относится к красной группе остановленной техники.

- `работает` - зеленый;
- `ожидание` - желтый;
- `простой`, `ремонт`, аварийная остановка и открытый критический простой - красный;
- красные и желтые единицы техники должны выводиться в приоритетные позиции активной смены, но красный имеет больший приоритет.

## Дополнение 24.06.2026: карточка техники по клику

В рабочем экране Горного диспетчера принято повторить механику Горного мастера: клик по любой единице техники открывает компактную карточку техники без перехода на другой экран.

Правило карточки:

- карточка открывается по клику, Enter и Space на плитке техники;
- карточка показывает тип техники, бортовой/гаражный номер, текущий статус, зону/комплекс, модель и ключевые рабочие сведения;
- если по технике есть открытая смена сотрудника, показывается бейдж текущего водителя или машиниста с телефоном;
- если сотрудник не назначен, в бейдже явно показывается `Сотрудник не назначен`;
- поля карточки не должны дублироваться: одинаковые лейблы выводятся только один раз;
- для демо-плиток, которые не связаны с реальной записью `Equipment`, карточка показывает минимальный демо-набор данных без подмены его реальной техникой.

Уточнение после визуальной проверки:

- карточка техники должна быть привязана к текущей теме пульта: дневная тема использует светлую панель с темным текстом, ночная тема - темную панель со светлым текстом;
- верхняя иконка в карточке не рисуется отдельным упрощенным компонентом, а повторяет гаражный слот техники со статусным контуром выполнения;
- если исходная плитка техники видна в гараже, карточка использует ее визуальный клон; если техника уже в комплексе, карточка собирает такой же слот из данных карточки;
- старый отдельный блок `.mm-detail-icon` для Горного диспетчера не используется, чтобы не было расхождения между гаражом и карточкой.

Дополнение по сменной информации:

- карточка техники должна показывать не только паспорт и текущий статус, но и мини-сводку смены на текущий момент;
- для самосвала выводятся рейсы, объем, активный рейс, количество экскаваторов и точек разгрузки, а также таблица `Экскаватор / Разгрузка / Порода / Рейсы / Объем / Последний`;
- для экскаватора выводятся рейсы, объем, количество самосвалов, забоев и разгрузок, а также таблица `Забой / Разгрузка / Порода / Самосвалы / Рейсы / Объем`;
- если в текущей базе для выбранной техники нет рейсов смены, допустим демонстрационный набор строк, чтобы оценить форму карточки до подключения полной отчетной логики.

Уточнение по визуальной форме:

- сменная информация в карточке должна восприниматься как мини-дашборд, а не только как электронная путевка;
- над табличной расшифровкой показывается графическое распределение объема горизонтальными барами;
- для самосвала график показывает распределение объема по направлениям `экскаватор / разгрузка / порода`;
- для экскаватора график показывает распределение объема по забоям `горизонт-блок / разгрузка / порода`;
- таблица остается ниже как точная расшифровка для проверки чисел.

Дальнейшая корректировка: вместо одинаковых горизонтальных гистограмм используются разные формы под смысл техники.

- самосвал показывает маршрутную схему `источник -> поток объема -> точка разгрузки`, где толщина/длина потока отражает долю объема;
- экскаватор показывает матрицу `забой -> ячейка разгрузки`, где цветная ячейка содержит точку разгрузки, объем и породу;
- таблица остается ниже как контрольная расшифровка.

Уточнение по визуальной выразительности дашборда:

- верхний графический блок карточки не должен ограничиваться горизонтальными гистограммами;
- добавлены круговые приборы доли объема по направлениям, чтобы карточка воспринималась как операторский мини-дашборд;
- сценарий `одна точка загрузки -> ККД + СКДР` должен явно отображаться и в маршрутах самосвала, и в матрице экскаватора;
- демонстрационная карточка самосвала показывает один комплекс, который в течение смены возил и на `ККД`, и на `СКДР`;
- демонстрационная карточка экскаватора показывает один забой, из которого отгрузка шла одновременно на `ККД` и `СКДР`.

## Принятая UX-логика

- основной формат роли: широкий desktop-пульт;
- mobile-first подход Горного мастера сюда не переносится как основной;
- интерфейс должен выглядеть как единый командный диспетчерский пульт, а не как обычная админская страница;
- верхняя строка отдается под статус смены, выполнение плана, руду, вскрышу, технику, тревоги и время;
- слева показывается парк экскаваторов и баланс самосвалов;
- в центре находятся производственные комплексы с кольцевыми индикаторами, планом, фактом, прогнозом, техникой и статусом;
- справа находятся приоритетные действия диспетчера и журнал событий;
- нижняя зона отдается под прогноз выполнения смены, потери и топ причин потерь;
- отдельный экран исключений нужен для служебных вмешательств: зависший рейс, неподтвержденное назначение, незакрытая смена, простой без причины, предупреждение отчета;
- каждое служебное действие должно иметь причину и попадать в журнал;
- ночной режим сохраняет ту же плотность данных, но снижает яркость фона и не меняет смысл статусных цветов.

## Визуальное направление после референсов

Принято ориентироваться на плотный темный диспетчерский dashboard:

- фон почти черный, панели темно-сине-зеленые;
- рабочие данные занимают почти весь экран без крупных пустых областей;
- ключевые статусы читаются цветом: зеленый - норма/работает, желтый - риск/ожидание, красный - критично/ремонт/дефицит;
- карточки комплексов крупнее обычных KPI-карточек и являются главным центром экрана;
- числовые показатели, проценты и тонны должны быть крупнее служебных подписей;
- действия диспетчера должны выглядеть как очередь приоритетов, а не как набор равнозначных кнопок;
- графики внизу нужны как быстрый прогноз и объяснение потерь, а не как декоративная аналитика.

Дополнение после визуальной критики: структура пульта сама по себе недостаточна. Для Горного диспетчера требуется уровень исполнения близкий к выбранному референсу: тонкие рамки, аккуратная типографика, стабильные приборные индикаторы, плотность без случайного шума и отсутствие ощущения схематичного wireframe.

## Состав первого набора экранов

### Пульт смены

Главная задача: за один взгляд показать, что происходит в смене и что требует решения до формирования отчета.

Ключевые блоки:

- объем смены;
- активные рейсы;
- неподтвержденные назначения;
- критические события;
- время до отчетной контрольной точки;
- комплексы и активные рейсы;
- очередь диспетчера;
- быстрые действия и переходы к отчетам.

### Исключения и коррекции

Главная задача: не смешивать обычное наблюдение со служебным исправлением данных.

Ключевые блоки:

- таблица проблем смены;
- тип события;
- объект события;
- состояние;
- ответственный;
- действие;
- карточка выбранного события;
- поле причины вмешательства;
- служебные действия с обязательной фиксацией в журнале.

### Ночной режим

Главная задача: сохранить читаемость диспетчерского пульта ночью и перед утренним отчетом.

Ключевые блоки:

- KPI ночной смены;
- события, которые нельзя потерять до 08:00;
- готовность суточного отчета;
- предупреждения перед Excel-выгрузкой.

## Границы роли

Горный диспетчер может:

- видеть всю сменную ситуацию;
- контролировать активные и завершенные рейсы;
- видеть неподтвержденные назначения;
- видеть незакрытые смены;
- выполнять служебные корректировки в рамках прав;
- открывать отчетную витрину и Excel-выгрузки;
- фиксировать причину вмешательства.

Горный диспетчер не должен:

- заменять Горного мастера в DnD-расстановке как основном сценарии;
- выполнять полевые действия без причины и журнального следа;
- видеть декоративные или маркетинговые блоки вместо плотного рабочего пульта;
- вручную собирать отчет из разрозненных таблиц, если данные уже есть в системе.

## Следующие решения

1. Визуально принять или скорректировать фрейм `GD / F / Эталонный polished dark-пульт`.
2. Уточнить, какие служебные действия войдут в первую демоверсию: завершить рейс, закрыть смену, снять назначение, исправить рейс, передать простой механику.
3. Определить, нужен ли отдельный экран `Рейсы` или достаточно очереди исключений плюс отчетной витрины.
4. После принятия Figma-логики подготовить перенос в Django отдельным этапом.

## Реализация 23.06.2026: первый Django-экран Горного диспетчера

В код перенесен первый вариант экрана `/dispatcher/control/` по концепции Figma `GD / K` и `GD / L`.

Сделано:

- основной первый viewport собран как плотная desktop-доска Горного диспетчера: верхние KPI, 12 плиток экскаваторов, 6 производственных комплексов, очередь действий, журнал, прогноз и потери;

Для следующей итерации рабочего экрана Горного диспетчера принято направление: уменьшить визуальный шум за счет отказа от отдельных кольцевых индикаторов процента над каждым комплексом и экскаватором. Отчетные блоки следует вынести на отдельный экран, оставив на основном пульте только оперативную сменную картину.

Техническое ограничение: текущие PNG-иконки техники нельзя точно использовать как stroke-progress по силуэту средствами CSS. ## Уточнение 24.06.2026: правило одного места для техники на доске диспетчера

Для рабочего прототипа `/dispatcher/control/` принято правило: единица техники может находиться только в одном месте интерфейса. Если экскаватор или самосвал назначен в комплекс, его плитка исчезает из соответствующего гаража. Если комплекс расформирован, экскаватор возвращается в левый гараж, а все самосвалы этого комплекса возвращаются в правый гараж.

Номер комплекса определяется номером экскаватора, а не ячейкой, куда пользователь бросил плитку. Экскаватор `1` формирует `K-1`, экскаватор `8` формирует `K-8`. Без экскаватора зона комплекса считается пустой; самосвалы не могут оставаться в комплексе без экскаватора.

Левый гараж экскаваторов не удерживает экран пустым пространством: когда все реальные экскаваторы переведены в активную смену, гараж схлопывается, а центральная область комплексов получает дополнительную ширину. При обратном перетаскивании комплекса влево гараж временно раскрывается как зона возврата, комплекс расформировывается, а экскаватор возвращается в свой слот.

Пустые будущие слоты экскаваторов не показывают номер, потому что в них позже может попасть техника с любым бортовым номером или названием. Статус выполнения сменного объема у реальных экскаваторов показывается по внешнему контуру самого слота; внутри слота остается только PNG-иконка и номер, без дополнительного внутреннего статусбара.

## Уточнение 24.06.2026: порядок активных комплексов и прорисовка пустых слотов

Пустые зоны активной смены не должны иметь названия `K-*` и подписи `свободная зона`: это просто свободные рабочие ячейки. Номер `K-*` появляется только у активного комплекса, когда в смену помещен экскаватор.

Активные комплексы занимают первые ячейки сетки и сортируются по номеру экскаватора. Если первым в смену добавлен экскаватор `6`, он временно занимает первую ячейку как `K-6`. Если затем добавлен экскаватор `5`, порядок становится `K-5`, `K-6`. Если затем добавлен экскаватор `8`, порядок становится `K-5`, `K-6`, `K-8`.

Остановленные/аварийные комплексы имеют приоритет показа и поднимаются в начало сетки, чтобы диспетчер видел проблему первой. После проблемных комплексов остальные активные комплексы снова идут по номеру экскаватора.

В левом гараже экскаваторов назначенный в смену экскаватор исчезает как техника, но его место остается прорисованным пустым слотом. Это нужно, чтобы визуально было понятно, куда возвращать экскаватор при расформировании комплекса. Если все реальные экскаваторы переведены в активную смену, левый гараж скрывается целиком и центральная сетка получает его ширину.

Пустой слот гаража не наследует статус ушедшего экскаватора. В нем не должно быть номера, цветного статуса, статусбара, тени или желтой/зеленой/красной подсветки. Пустое место отображается как нейтральная парковочная ячейка с приглушенным силуэтом экскаватора. Статусный цвет, номер и прогресс по контуру показываются только тогда, когда сам экскаватор физически находится в гараже.
## Реализация 24.06.2026: фиксированные парковочные места экскаваторов в гараже диспетчера

В Django-прототипе `/dispatcher/control/` левый гараж экскаваторов переведен из обычного списка в фиксированную вертикальную стойку парковочных мест. Каждая реальная машина имеет свой физический слот по бортовому номеру: экскаватор `1` занимает строку `1`, экскаватор `8` занимает строку `8`. Если экскаватор перетаскивается в активную смену, его строка не схлопывается и остальные машины не поднимаются вверх; на месте остается нейтральная парковочная ячейка без номера, статуса, тени и статусбара. При расформировании комплекса экскаватор возвращается в свой исходный слот.

Статусбар выполнения плана у экскаватора в гараже теперь рисуется по квадратному контуру самого слота через CSS `::before` и `conic-gradient`. У пустых будущих слотов и у слотов, из которых экскаватор назначен в активную смену, этот слой отключается. Внутри реального слота остаются только проектная PNG-иконка экскаватора и номер.

Сортировка активной смены уточнена: первыми выводятся все проблемные комплексы, сначала красные/остановленные, затем желтые/риск, после них рабочие комплексы в порядке номера экскаватора. Проверенный пример стартового состояния: `K-6` как остановленный, затем `K-3` как риск, затем `K-1`, `K-2`, `K-4`, `K-5` как нормальные.

## Уточнение 24.06.2026: разделены карточки комплекса, экскаватора и самосвала

В `/dispatcher/control/` принято разделение владельцев информации, чтобы карточки не дублировали друг друга.

Карточка комплекса отвечает за оперативную связку `экскаватор + назначенные самосвалы`: статус комплекса, план/факт/прогноз, процент сменного выполнения, текущую породу, баланс транспорта, причину риска/остановки и диспетчерское действие. Карточка самосвала отвечает за транспортную единицу: водитель, статус, текущий комплекс или гараж, рейсы за смену, объем за смену, комплексы, породы, точки разгрузки, средний цикл и ожидание разгрузки. Для самосвала остаются вкладки `Маршруты`, `По разгрузке`, `По породе`, `По комплексам`.

Технически активные зоны комплекса получили собственный `card_id` вида `complex-K-*`; клик по комплексу теперь открывает карточку типа `Комплекс`, а не карточку экскаватора. Исходная связь с экскаватором сохранена отдельно в `data-source-equipment-card-id`, чтобы не смешивать смысл карточки и механику DnD.

## Дополнение 24.06.2026: Figma-набор экранов диспетчерской аналитики

После обсуждения принято разделить рабочее место диспетчерской на несколько связанных экранов, а не перегружать текущий пульт смены полноценными отчетами. Текущий `Пульт` остается экраном оперативного управления сменой и DnD-расстановки техники.

В Figma-файле `Горный мастер - mobile UX concept` на странице `UX-kit / Горный диспетчер` создан первый темный набор экранов в стиле текущего диспетчерского пульта:

Уточнение по подключению к данным: экраны `Горные объемы`, `Простои`, `Журнал` и часть `Отчетов` могут строиться на текущих моделях `Trip`, `EmployeeShift`, `DowntimeEvent`, `DispatcherActionLog`, `ReportTemplate` и существующих Excel-экспортах. - `Вариант L / Двухстрочная шапка по референсу с нашим бейджем`.
- `Вариант O / Иконки навигации + встроенная правая зона часов`.
- `Вариант Q / Командная панель с табами и круглыми часами`.

Общие принципы:

- рабочая зона главного пульта ниже шапки не затрагивается;
- верхняя навигация должна быть стабильной на всех диспетчерских экранах;
- кнопки основных разделов отделяются от служебных кнопок `Назад` и `Выйти`;
- бейдж диспетчера должен показывать человека и состояние смены, а не спорные KPI;
- день/ночь оформляется отдельной пиктограммой солнца или месяца;
- сменное действие `Начать смену` / `Завершить смену` остается отдельным элементом с повышенным визуальным весом для завершения смены.

Уточнение: предпочтительное направление для дальнейшей проработки - вариант D. В нем крупный бейдж действующего диспетчера с фото/инициалами размещен слева, а кнопки навигации, `Назад`, `Выйти`, переключатель темы и сменное действие приведены к общей геометрии кнопок. Отличие между ними задается ролью и цветом: активный раздел зеленый, завершение смены красное, служебные действия нейтральные.

Уточнение 2: более правильное направление - вариант G. В нем бейдж не вставляется в шапку как отдельная карточка, а сама левая часть шапки является бейджем активной смены: фото/инициалы, имя, роль и состояние смены встроены в общую верхнюю панель. Крупный заголовок `Горный диспетчер` убран, чтобы не дублировать роль из бейджа; слева после зоны смены показывается текущий экран, например `Пульт смены`.

Уточнение 3: актуальное направление - вариант H. В нем левая часть шапки остается встроенным бейджем активной смены, но фото/инициалы сделаны крупнее и оформлены квадратом со скругленными углами в геометрии кнопок. Под названием текущего экрана показывается название смены (`Дневная смена` / `Ночная смена`), а не дата и время смены, чтобы не дублировать правую временную зону. Текущее время и дата справа оформлены как встроенное поле шапки с крупными цифрами, а не как отдельный слот-кнопка.

Уточнение 4: текущий лучший кандидат - вариант L. Он использует двухстрочную структуру по референсу пользователя: в верхней строке текущий экран, служебные действия и время; в нижней строке навигация по разделам и отдельное сменное действие. Это убирает проблему равнозначной линейки кнопок: `Пульт` / `Аналитика` / `Журнал` / `Отчеты` остаются группой навигации, `Завершить смену` отделено как операционное действие, а `День/ночь`, `Назад`, `Выйти` остаются служебной группой. Левый бейдж остается в авторском стиле проекта.

Уточнение 5: после дополнительных референсов появились два сильных направления. Вариант O сохраняет крупный левый бейдж, использует вертикальные плитки разделов с иконками и правую встроенную зону времени на всю высоту шапки. Вариант Q развивает третий референс: центральная навигация сделана как табы с иконками и нижними индикаторами, а справа используются круглые часы. O более строгий и ближе к текущей компоновке, Q визуально богаче и функционально выразительнее, но выше и сложнее.

## Реализация 25.06.2026: перенос варианта Q в Django без уменьшения пульта

В общий include `includes/dispatcher_header.html` перенесена компоновка направления Q: крупный левый бейдж активного диспетчера, центральный контекст смены, навигационные табы с CSS-иконками, отдельная зона сменного действия, служебные кнопки и правые круглые часы.

Ключевое ограничение после проверки: высота строки шапки главного пульта не увеличивается. Рабочая зона `/dispatcher/control/` остается в прежней сетке `clamp(108px, 11vh, 126px)`; все декоративные элементы шапки ужимаются внутрь этой высоты.

Круглая шкала часов реализована SVG-кольцами с квадратным контейнером `104 x 104`, чтобы шкала не была овальной или смещенной. Проверено, что кольцо находится внутри шапки, а навигация сохраняет одинаковые координаты на `/dispatcher/control/`, `/dispatcher/management/`, `/dispatcher/shift-log/`, `/dispatcher/reports/`.

Дополнительно убран лимит ширины `1760px` у `/dispatcher/management/`, из-за которого шапка витрины сдвигалась относительно остальных диспетчерских экранов.

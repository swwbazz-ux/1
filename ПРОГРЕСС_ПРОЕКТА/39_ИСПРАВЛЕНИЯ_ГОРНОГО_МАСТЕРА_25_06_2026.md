# Исправления Горного мастера 25.06.2026

## Карточки техники

На desktop-экране `/mining-master/assignments/` исправлена механика вызова карточек техники.

Причина сбоя: обычный `click` был отключен для draggable-плиток, поэтому карточка открывалась только через побочную mouse-DnD-логику. Из-за этого часть плиток техники могла не открывать карточку при обычном клике.

Что изменено:

- единый обработчик клика теперь работает для всех плиток с `data-equipment-id`;
- после настоящего DnD клик подавляется, чтобы карточка не открывалась сразу после перетаскивания;
- для плиток экскаваторов в левом гараже добавлен явный `data-equipment-card-id`;
- пустой активный экскаватор больше не получает статус `Резерв`; статус карточки заменен на `Без сам.`.

Проверено:

- `manage.py check`;
- `manage.py test assignments` - 15 тестов;
- Browser QA `/mining-master/assignments/`: карточка открывается по клику у экскаватора в левом гараже, у самосвала в правом гараже и у зоны комплекса;
- ошибок консоли нет;
- статус `Резерв` в данных карточек отсутствует.
## Исправление 25.06.2026: desktop-DnD горного мастера без системных alert

На `/mining-master/assignments/` добавлен собственный UX-попап для ошибок действий вместо браузерного `window.alert`.

Дополнительно старый мобильный DnD-слой ограничен так, чтобы на desktop-пульте он не вмешивался в новую серверную механику:

- невозможные drop-действия на desktop больше не должны запускать старые обработчики;
- после неудачного или прерванного перетаскивания очищаются ghost-карточки и подсветки drop-зон;
- случайное выделение текста на рабочей dispatcher-оболочке отключено через существующее `user-select: none`;
- при закрытой смене элементы не получают `data-dispatcher-drag`, поэтому редактирование пульта заблокировано.

Проверено:

- `manage.py check`;
- `manage.py test assignments` - 17 тестов;
- Browser QA после перезапуска `START_SERVER_MVP.bat`: `/mining-master/assignments/` отдает новый DOM с `data-mining-master-notice`, `user-select: none`, без горизонтального/вертикального overflow и без системных browser-dialogs; при закрытой смене `draggable: 0`.

### Уточнение 26.06.2026

- После проверки в Chrome обнаружено, что при старом значении сессии `device_kind=personal` диспетчерский пульт мог показывать обычный confirm `Начать смену горного диспетчера?` вместо попапа входа.
- Исправлено: для desktop-пульта Горного диспетчера при закрытой смене `requires_shift_reauth` теперь включается всегда, независимо от старого значения `device_kind` в сессии.
- Серверный обработчик старта смены дополнительно блокирует POST `shift_action=start` без `reauth_phone` и `reauth_access_code`, чтобы старый прямой confirm не мог открыть смену.
- Добавлен тест на случай `device_kind=personal`: `/dispatcher/control/` все равно рендерит `data-shared-shift-login-open`.

### Уточнение 26.06.2026 по маске кода

- Поле `Код мастера` / `Код диспетчера` в общем попапе смены теперь вводится в том же виде, что и подсказка: `00-00-00`.
- HTML-поле использует `maxlength=8` и pattern `[0-9]{2}-[0-9]{2}-[0-9]{2}`.
- JS маска форматирует ввод и вставку из буфера в группы по две цифры.
- Серверная проверка нормализует код обратно в 6 цифр перед сравнением с `EmployeeAccess.access_code`, поэтому старые коды в базе менять не нужно.

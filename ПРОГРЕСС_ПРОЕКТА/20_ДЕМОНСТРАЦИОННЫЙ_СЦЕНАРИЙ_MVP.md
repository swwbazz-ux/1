# Демонстрационный сценарий MVP производственного контура

Дата создания: 17.06.2026

## Цель сценария

Показать первый рабочий цикл системы без лишней теории:

1. сотрудник входит в единое приложение по своему коду доступа;
2. система открывает интерфейс по роли;
3. водитель открывает смену;
4. горный мастер назначает самосвал под экскаватор;
5. водитель подтверждает назначение;
6. машинист экскаватора создает рейс;
7. водитель завершает рейс;
8. диспетчер видит процесс на пульте;
9. отчет по объемам собирается автоматически и выгружается в Excel.

Главная мысль показа: данные больше не собираются вручную в конце смены, а появляются в системе там, где реально возникает событие.

## Участники демонстрации

### Администратор

Задача: показать, что система управляется централизованно.

Что можно показать:

- сотрудники;
- роли;
- доступы;
- техника;
- породы;
- точки разгрузки;
- правила кубатуры;
- шаблоны отчетов.

### Водитель самосвала

Задача: показать минимальное количество действий водителя.

Сценарий:

1. водитель входит по своему коду;
2. при первом входе выбирает смену, самосвал и место проживания;
3. открывает смену;
4. подтверждает назначение кнопкой `Принял`;
5. видит активный рейс;
6. после разгрузки нажимает `Выполнено`;
7. закрывает смену, указав топливо, пробег и моточасы на конец.

Что важно проговорить:

- водитель не считает объемы;
- водитель не выбирает сложные параметры отчета;
- система сама фиксирует время событий;
- если рейс остается активным на пересменке, смена все равно может быть закрыта.

### Горный мастер

Задача: показать расстановку техники.

Сценарий:

1. горный мастер входит по своему коду;
2. выбирает экскаватор;
3. выбирает самосвал;
4. создает назначение;
5. водитель видит это назначение у себя.

Что важно проговорить:

- горный мастер формирует рабочую связку `экскаватор - самосвал`;
- водитель не назначает себе технику сам после первичной регистрации;
- назначение становится рабочим после подтверждения водителем.

### Машинист экскаватора

Задача: показать создание рейса там, где начинается груз.

Сценарий:

1. машинист входит по своему коду;
2. видит принятые назначения самосвалов;
3. выбирает самосвал;
4. выбирает породу;
5. выбирает точку разгрузки;
6. отправляет самосвал в рейс.

Что важно проговорить:

- объем рейса считается автоматически;
- тоннаж считается автоматически, если в справочнике породы есть плотность;
- рейс попадает водителю как активный;
- рейс сразу становится видимым диспетчеру.

### Диспетчер

Задача: показать, что диспетчер видит процесс, а не собирает его вручную.

Сценарий:

1. диспетчер входит по своему коду;
2. открывается диспетчерский пульт;
3. видит активные незакрытые рейсы;
4. видит назначения, ожидающие подтверждения;
5. видит принятые назначения;
6. видит последние выполненные рейсы;
7. фильтрует пульт по самосвалу или экскаватору;
8. открывает отчет по объемам.

Что важно проговорить:

- диспетчер не переписывает данные из разных источников;
- пульт нужен для контроля смены в процессе;
- отчет нужен для итогов и выгрузки.

### Руководство

Задача: показать управленческую ценность.

Что можно показать на текущем этапе:

- общий объем по выполненным рейсам;
- общий тоннаж;
- выполненные рейсы;
- переходящие рейсы;
- фильтры по сменам;
- выгрузку в Excel.

Что пока нужно подготовить отдельно:

- отдельную страницу руководителя с красивыми показателями;
- графики;
- суточную витрину по заказчику;
- сравнение плана и факта, если появится план.

## Главный демонстрационный цикл

### Шаг 1. Вход водителя

Код демо-доступа: `2000`

Ожидаемый результат:

- открывается интерфейс водителя;
- если первичная регистрация уже пройдена, водитель видит смену;
- если смена закрыта, можно открыть новую.

### Шаг 2. Вход горного мастера

Код демо-доступа: `4000`

Ожидаемый результат:

- открывается экран назначения самосвала под экскаватор;
- создается назначение;
- у водителя появляется уведомление о назначении.

### Шаг 3. Подтверждение водителем

Ожидаемый результат:

- водитель нажимает `Принял`;
- назначение становится принятым;
- его можно использовать для создания рейса.

### Шаг 4. Вход машиниста экскаватора

Код демо-доступа: `3000`

Ожидаемый результат:

- машинист создает рейс;
- выбирает породу и точку разгрузки;
- система автоматически считает объем и тоннаж.

### Шаг 5. Пульт диспетчера

Код демо-доступа: `5000`

Ожидаемый результат:

- диспетчер видит активный рейс;
- может отфильтровать по самосвалу или экскаватору;
- после завершения рейса видит его в последних выполненных.

### Шаг 6. Завершение рейса водителем

Ожидаемый результат:

- водитель нажимает `Выполнено`;
- рейс становится выполненным;
- рейс появляется в отчете.

### Шаг 7. Отчет по объемам

Ожидаемый результат:

- отчет показывает объемы;
- отчет показывает тоннаж;
- отчет показывает смену загрузки и смену разгрузки;
- переходящий рейс помечается отдельно;
- отчет можно выгрузить в Excel;
- если выбран шаблон отчета, выводятся только столбцы шаблона.

## Подготовка демо-данных

Для подготовки демонстрационной базы нужно выполнить команду:

```text
python manage.py seed_demo_scenario
```

Команда создает роли, демо-доступы, смены, назначение, активный рейс, выполненный рейс и шаблон отчета.

После выполнения доступны коды:

- `1000` - администратор;
- `2000` - водитель;
- `3000` - машинист экскаватора;
- `4000` - горный мастер;
- `5000` - диспетчер.

## Критерий готовности к первому показу

Первый показ можно считать готовым, если без ручного вмешательства в базу можно пройти цикл:

`водитель -> горный мастер -> водитель -> машинист экскаватора -> водитель -> диспетчер -> отчет -> Excel`.

Если этот цикл проходит стабильно, система уже демонстрирует главную ценность MVP.

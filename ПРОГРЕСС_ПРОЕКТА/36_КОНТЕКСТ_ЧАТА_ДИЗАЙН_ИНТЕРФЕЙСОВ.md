# Контекст заблокированного чата "Дизайн интерфейсов"

Дата восстановления: 23.06.2026.

Заблокированный чат: `Дизайн интерфейсов`, thread id `019ee576-2ee0-7fd2-a168-bcf84b5e1acd`, статус `systemError`.

Причина блокировки по восстановленной истории: блокировка возникла не из-за дизайна интерфейсов, а после запроса создать отдельный файл со всеми логинами и паролями/пинкодами для MVP. Дальше этот чат лучше не использовать для хранения или генерации файлов с доступами. Если нужны текущие тестовые входы, безопаснее просить показать их прямо в чате или проверять через локальную базу без создания отдельного файла с паролями.

## Главная выжимка

В чате была доведена рабочая админка MVP и часть связанных решений по серверу. Основная цепочка работ:

1. Карточка сотрудника стала визуальным эталоном для админки.
2. По карточке сотрудника был собран экран создания сотрудника.
3. В код были перенесены экраны админки: справочники, карточка справочника, конфликты, журнал действий, выгрузки.
4. Выполнена финальная техническая сверка админки.
5. После этого обсуждалась серверная инфраструктура и выбор HOSTKEY для демонстрационного переноса.
6. Позже в этом же чате исправлялся баг справочника техники со сбросом прокрутки после сохранения.

## Карточка сотрудника

Принятые решения:

- карточка сотрудника является базовым визуальным образцом для остальных экранов админки;
- ФИО разделено визуально на `Фамилия`, `Имя`, `Отчество`;
- `Телефон` находится в верхнем блоке;
- `Табельный номер` был заменен на `Вахта`;
- лишняя нижняя `Вахта` и пустой служебный блок убраны;
- фото должно быть квадратным, без отдельного видимого контейнера вокруг фото;
- сторона фото в финальной правке уменьшена до `235px`;
- все однострочные `input/select` в админских формах сотрудника приведены к высоте `46px`;
- размер шрифта в полях приведен к `14px`;
- клик по существующему фото открывает попап просмотра;
- загрузка/замена фото выполняется через `+` под фото;
- удаление фото выполняется через `-` под фото;
- при переносе в код отдельные визуальные поля ФИО и проживания собираются обратно в существующие поля модели через JS, без миграций.

Проверки, которые фиксировались в чате:

- `manage.py check`;
- `manage.py test users`;
- перезапуск через `STOP_SERVER_MVP.bat` и `START_SERVER_MVP.bat`;
- браузерная проверка карточки сотрудника на desktop/mobile.

## Серверный контекст из этого чата

Обсуждался будущий перенос MVP на сервер:

- текущая локальная БД: `СИСТЕМА_MVP/backend/db.sqlite3`;
- на сервере нужен PostgreSQL, не SQLite;
- для демо производственного контура выбран HOSTKEY `vm.v2-medium`;
- характеристики выбранного демо-тарифа: `8 vCPU / 16 GB RAM / 160 GB NVMe`, IPv4, `1 Gbit/s`, `3 TB`;
- цена-ориентир: около `3390 руб/мес`;
- расчет демо-нагрузки: около `72 000 рейсов за месяц`;
- реальный перенос и серверные команды были переданы в отдельный чат `Перенос на сервер`.

## Как продолжать после блокировки

1. Не продолжать работу в заблокированном чате `Дизайн интерфейсов`.
2. Для текущей работы использовать этот чат или новый рабочий чат, предварительно читая проектные файлы прогресса.
3. Если нужны доступы MVP, не просить создать отдельный файл с логинами/паролями. Формулировка безопаснее: `Покажи текущие тестовые входы MVP из локальной базы: роль, ФИО, телефон, пинкод, статус доступа. Файл не создавай.`
4. Для UI-работ использовать проектные skills:
   - `accounting-system-ux-review`;
   - `accounting-system-design-system`;
   - `accounting-system-browser-qa`.
5. Для browser QA запускать сервер только через `START_SERVER_MVP.bat`, останавливать только через `STOP_SERVER_MVP.bat`.
6. Не откатывать старые изменения в рабочем дереве: в чате фиксировалось, что там есть много старых незавершенных изменений по админке и Горному мастеру.

## Что подтверждено исходными документами

Источник кадровой истины: `Список сотрудников компании.xlsx` из выгрузки 1С.

- 479 строк сотрудников;
- 41 уникальная официальная должность;
- должность хранится в 1С отдельно от фактической расстановки на технику;
- расстановочные файлы отражают временную производственную работу внутри вахты и не должны переписывать кадровую должность.

Главное подтверждение: официальная должность `Водитель автомобиля грузового` не равна водителю технологического самосвала. В рабочем документе `заявка на билеты на вахту14.07.2026.xlsx`, лист `Числ`, она разделена на:

| Производственная специализация | Численность на 14.07.2026 |
|---|---:|
| Технологический самосвал | 83 |
| Технологический самосвал NHL | 20 |
| ПРМ | 3 |
| Топливозаправщик | 5 |
| КДМ | 4 |
| Вахтовый автобус | 5 |
| Передвижная столовая на базе КАМАЗ | 0 |

Технологические самосвалы и NHL в этом документе идут отдельными строками от общей должности `Водитель автомобиля грузового`. Поэтому их нельзя автоматически смешивать с КДМ, ПРМ, АТЗ и автобусом только из-за слова «водитель».

## Единые понятия

| Понятие | Что означает | Кто изменяет |
|---|---|---|
| **Кадровая должность** | Официальная должность из 1С и кадрового приказа. У сотрудника всегда одна. | ОУП |
| **Базовая производственная специализация** | Что сотрудник реально может делать по своей кадровой должности: технологический самосвал, КДМ, ПРМ, экскаватор и т.д. Для однозначных должностей подставляется автоматически; для широких выбирается ОУП. | ОУП |
| **Эффективная специализация** | Базовая специализация либо временная, пока действует подтвержденный перевод. Именно она определяет список техники и рабочее приложение. | Рассчитывает сервер |
| **Роль доступа** | Технический доступ в одно конкретное приложение: водитель, машинист экскаватора, ОУП, диспетчер и т.д. Это не кадровая должность. | Сервер по специализации или администратор для управленческих ролей |
| **Назначение на смену и технику** | Оперативная расстановка на конкретную смену. Она не меняет должность и не выдает приложение. | Заместитель начальника участка |

Поле `Рабочая категория` в его нынешнем виде (`driver`, `excavator_operator`, `other`) недостаточно: оно смешивает должность, специализацию и приложение. В целевой карточке его заменяют видимые человеку поля `Кадровая должность` и `Производственная специализация`; техническая роль доступа не дублируется как кадровое поле.

## Минимальная целевая модель

Нужны три новых справочных/исторических слоя, а не новый набор полей для каждой роли.

### 1. Справочник кадровых должностей

`PersonnelPosition`:

- код и название должности из 1С;
- признак, требуется ли выбор специализации;
- разрешенные специализации;
- базовая специализация по умолчанию для однозначных должностей;
- `is_active` только для управления справочником: неактивную строку нельзя выбрать для новой карточки, но история сотрудников не пропадает.

### 2. Справочник производственных специализаций

`ProductionSpecialization`:

- код и понятное название;
- подходящий вид техники или группа техники, если она нужна;
- роль доступа/приложение либо пустое значение, если приложения еще нет;
- `is_active` только для управления справочником.

Примеры: `Технологический самосвал`, `КДМ`, `ПРМ`, `Топливозаправщик`, `Вахтовый автобус`, `Машинист экскаватора`, `Бульдозер`, `Автогрейдер`.

У сотрудника хранится одна кадровая должность и одна базовая специализация. Для однозначных должностей базовая специализация выбирается сервером; для широких ОУП обязан выбрать ее из допустимого списка.

### 3. Временный производственный перевод

`TemporaryWorkTransfer`:

- сотрудник;
- исходная и целевая специализации snapshot;
- связанная `WatchPeriod`;
- дата/время вступления в силу и дата/время окончания;
- статус: `requested`, `approved`, `rejected`, `cancelled`, `expired`;
- кто запросил, кто утвердил, причина и комментарий ОУП;
- время создания, решения и завершения.

У сотрудника может быть только один действующий временный перевод. Кадровая должность при таком переводе не меняется.

В модели уже есть `shifts.WatchPeriod` с `starts_on`, `ends_on` и `is_active`. Сейчас она почти не участвует в расстановке, поэтому временный перевод должен всегда ссылаться на конкретную вахту и вычислять завершение из `ends_on`. Даты нельзя брать из свободного текстового поля `Employee.rotation`.

## Рабочий процесс

```text
1. ОУП создает или редактирует кадровую карточку
   -> выбирает официальную должность
   -> при необходимости выбирает базовую специализацию
   -> сервер определяет доступное приложение

2. Заместитель начальника участка открывает расстановку
   -> выбирает технику или группу техники
   -> видит только сотрудников с подходящей эффективной специализацией

3. Нужен сотрудник из другой группы
   -> заместитель создает заявку на временный перевод до конца активной вахты
   -> ОУП подтверждает или отклоняет
   -> при подтверждении меняется только эффективная специализация

4. До конца вахты
   -> сотрудник виден только в подходящей новой группе
   -> старое приложение недоступно, новое приложение доступно
   -> PIN и пароль сохраняются, если у сотрудника уже был доступ

5. Конец WatchPeriod
   -> сервер перестает считать перевод действующим даже до фоновой очистки записи
   -> возвращается базовая специализация и исходное приложение
   -> запись получает статус expired и остается в истории
```

Если исходная специализация не имела приложения, а временная или кадровая новая специализация его требует, доступ создается при подтверждении. Это единственный случай, когда сохраненного прежнего PIN физически нет.

Перевод нельзя включать поверх открытой смены или действующего назначения по прежней группе: сначала сотрудник должен быть освобожден из текущей расстановки. Уже открытая смена остается историческим snapshot и не переписывается задним числом.

## Инварианты

- один сотрудник — одна кадровая должность;
- одна базовая и одна эффективная производственная специализация;
- одно активное рабочее приложение на номер телефона и PIN;
- заместитель не может назначить человека на технику, если эффективная специализация не подходит;
- ОУП не может утвердить временный перевод дальше `WatchPeriod.ends_on`;
- после окончания вахты сотрудник не остается в чужой группе даже при задержке фоновой задачи;
- ежедневный `CrewPlan` и `EquipmentAssignment` остаются расстановкой, а не источником кадровой квалификации;
- история должности, перевода, доступа и назначения не удаляется.

## Миграция без ложных назначений

1. Справочник должностей заполняется точными 41 значениями из 1С.
2. Однозначные сотрудники получают базовую специализацию автоматически: технологический самосвал и машинист экскаватора.
3. 36 сотрудников с должностью `Водитель автомобиля грузового` не получают приложение самосвала автоматически. Они попадают в контролируемый список «требуется специализация» до выбора ОУП.
4. Существующие активные доступы не удаляются молча. Для каждой неоднозначной карточки формируется сверка: кадровая должность, текущая категория, текущий доступ, необходимое действие.
5. После сверки старое поле `work_category` становится только совместимым техническим источником на переходный период и больше не используется как основное правило допуска к технике.

## Вопросы, которые нельзя угадывать

1. Нужны ли отдельные квалификационные ограничения между экскаваторами 3000, 4000 и планировщиком. Текущая задача про планы техники не доказывает, что машинист 3000 не может быть назначен на 4000; пока это не подтверждено, у всех остается одна специализация `Машинист экскаватора`.
2. Какие из механиков действительно должны получить приложение механической службы. Должность является кандидатом, но не автоматическим допуском.
3. Как именно выбирается активный `WatchPeriod` для разных вахт/бригад. В справочнике уже есть даты периода, но текущая расстановка использует текст `rotation` и не связывается с `WatchPeriod`.

- Поле `Статус сотрудника` одинаково отображается только для чтения на обоих экранах; статус трудовых отношений меняется отдельными действиями деактивации, архива, увольнения и восстановления.
- Первичная дата приема в обоих контурах заполняется текущей датой, но может быть исправлена до сохранения.
- Основное действие на обоих экранах называется `Создать сотрудника`. Права остаются разными: ОУП не назначает смену и технику, а видит, что расстановку выполняет заместитель начальника участка.

# План этапа 2. Технический каркас ядра

Дата создания: 17.06.2026

## Назначение документа

Этот документ переводит проект из этапа проектирования MVP в этап создания технического каркаса ядра.

Этап 1 дал понимание, что строим. Этап 2 должен дать рабочую техническую основу, на которую потом будут подключаться интерфейсы водителя, машиниста экскаватора, горного мастера, диспетчера, админки и отчетов.

## Цель этапа 2

Создать минимальное ядро системы:

- структура проекта;
- база данных;
- модели основных сущностей;
- роли и доступы;
- загрузка стартовых справочников;
- базовая админка;
- единый вход;
- техническая основа для первых экранов MVP.

## Предварительный стек

Для первого технического каркаса используем:

- Python;
- Django;
- PostgreSQL;
- HTML;
- CSS;
- TypeScript/JavaScript;
- PWA позже, после появления первых рабочих экранов.

На локальном этапе допускается использовать SQLite для быстрой проверки каркаса, но целевая база для нормальной системы - PostgreSQL.

## Предлагаемая структура проекта

В рабочей папке проекта нужно создать отдельную папку для кода, например:

`СИСТЕМА_MVP`

Внутри нее:

- `backend` - серверная часть Django;
- `frontend` - будущие пользовательские интерфейсы, если будет выбран отдельный фронтенд;
- `data_import` - скрипты и файлы загрузки справочников;
- `docs` - техническая документация для разработчика;
- `exports` - тестовые выгрузки отчетов;
- `local_data` - локальные рабочие данные, которые не должны считаться эталонными.

На первом шаге можно начать с Django-проекта в `backend`, а интерфейсы временно делать внутри Django, чтобы быстрее получить рабочую систему.

На этапе 2 нужно подготовить загрузку уже найденных CSV:

Важно: справочник экскаваторов нужно сверить, потому что у компании 8 собственных экскаваторов, а найденные данные были неполными.

## Минимальный первый прототип

Первый технический прототип должен доказать, что ядро работает.

Минимальный сценарий:

1. Администратор создает сотрудника.
2. Администратор выдает сотруднику роль и доступ.
3. Сотрудник входит в систему.
4. Система открывает интерфейс по роли.
5. Водитель проходит первичную регистрацию.
6. Система проверяет занятость самосвала.
7. Водитель открывает смену.
8. Водитель подтверждает топливо, пробег и моточасы.
9. Горный мастер назначает самосвал под экскаватор.
10. Водитель нажимает `Принял`.
11. Машинист экскаватора создает рейс.
12. Водитель завершает рейс кнопкой `Выполнено`.
13. Система показывает рейс в диспетчерском контроле.
14. Система формирует простой отчет по объемам.

## Что не делаем в самом начале этапа 2

Чтобы не распылиться, в начале этапа 2 не делаем:

- красивый финальный дизайн;
- полноценный PWA;
- RuStore-приложение;
- полный конструктор отчетов;
- сложную аналитику;
- полный модуль механиков;
- полный кадровый контур;
- полный модуль проживания.

Сначала нужно, чтобы заработало ядро данных.

## Критерии готовности этапа 2

Этап 2 можно считать готовым, если:

- создана структура проекта;
- Django-проект запускается локально;
- есть база данных;
- есть модели основных сущностей;
- доступна админка;
- можно создать сотрудника, роль и доступ;
- можно загрузить стартовые справочники;
- можно войти в систему;
- система понимает роль пользователя;
- есть техническая основа для экранов MVP;
- создан первый тестовый рейс в базе;
- первый простой отчет по объемам строится из данных рейсов.

## Риски этапа 2

- неполные справочники экскаваторов;
- возможные расхождения в текущих Excel-формах отчетов;
- необходимость быстро выбрать PostgreSQL или временно стартовать на SQLite;
- риск слишком рано уйти в дизайн вместо ядра;
- риск добавить лишние поля в рабочие интерфейсы.

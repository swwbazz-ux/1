# Ручные сменные планы техники

Дата фиксации: 04.07.2026.

## Причина

На текущем этапе системе нужен не только учет фактических рейсов, но и план, с которым этот факт сравнивается.

Будущий модуль сможет автоматически формировать сменные планы по технике, комплексам и производственной задаче. До появления такого модуля администратор должен вручную задавать планы на самосвалы и экскаваторы через админку.

## Что принято

В MVP вводится ручной контур сменного планирования:

- общий план объема в кубах;
- тип общего плана: месячный, суточный, дневная смена или ночная смена;
- план по конкретной единице техники;
- план рейсов только для техники, где выполнение считается рейсами, например самосвалы;
- план объема, м3 для техники, где выполнение считается объемом, например экскаваторы;
- правило расчета выполнения техники: по рейсам или по объему;
- активность плана;
- комментарий администратора.

Общий план не должен заставлять администратора создавать отдельную запись на каждую календарную дату и смену.
Дата и расчетная смена могут оставаться техническими полями базы для совместимости отчетов, но в рабочей админке они не должны быть основным способом управления планами.

План в тоннах на текущем этапе не используется в рабочей админке. Тоннаж может оставаться фактическим отчетным показателем, но не является плановым полем первого рабочего контура.

## Где задается

В рабочей админке MVP добавляются разделы:

- `Сменные планы`;
- `Планы техники`.

Они находятся в разделе справочников админки:

- `/system-admin/references/shift-plans/`;
- `/system-admin/references/equipment-shift-plans/`.

## Как это работает сейчас

Администратор создает общий план объема:

- месячный план в м3;
- суточный план в м3;
- дневной сменный план в м3;
- ночной сменный план в м3.

Затем при необходимости задает отдельные планы по самосвалам и экскаваторам:

- для самосвалов NHL и БелАЗ план обычно задается в рейсах;
- для экскаваторов план задается в м3.

Если по сменному плану есть планы конкретной техники, итоговый план смены собирается из них.

Если планов конкретной техники нет, система использует общий сменный план объема.

## Важное UX-ограничение

Серверу все равно нужны дата и смена для расчета факта, отчетов и прогресса, потому что завершенные рейсы относятся к конкретным сменам.

Но рабочий интерфейс администратора должен оперировать понятными плановыми сущностями:

- месяц;
- сутки;
- дневная смена;
- ночная смена;
- техника.

Следующий слой интерфейса должен дать администратору применение планов на период или вахту и исключения по отдельным датам, но базовая форма не должна выглядеть как ежедневное ручное заполнение одинаковых планов.

## Что считается сервером

Сервер считает:

- факт рейсов;
- факт объема;
- факт тоннажа;
- процент выполнения плана;
- прогресс конкретной техники за смену.

Для выполнения общего сменного плана в расчет смены попадают только планы типов `Дневная смена` и `Ночная смена`.
`Суточный план` и `Месячный план` являются отдельными плановыми ориентирами и не должны автоматически складываться со сменными планами, чтобы не получить двойной счет.

Для самосвала факт считается по завершенным рейсам разгрузки.

Для экскаватора факт считается по завершенным рейсам загрузки.

## Почему экран руководства мог быть пустым

Витрина руководства не должна считать незавершенный рейс как готовый объем.

Если экскаваторщик отправил самосвал на разгрузку, но водитель еще не подтвердил выгрузку, рейс находится в работе. В этом состоянии у него еще нет финального объема и тоннажа для отчетности.

Рейс попадает в управленческие итоги только после завершения разгрузки, когда сервер фиксирует:

- `status = completed`;
- `volume_m3`;
- `tonnage`;
- смену загрузки;
- смену разгрузки.

Поэтому при имитации рейса экран руководства может не показать объем, если рейс еще активный или завершен другой датой.

## Что изменилось для витрины руководства

Витрина руководства теперь должна брать план на день из ручных сменных планов админки.

Если ручного плана на дату нет, используется прежний резервный источник: плановые значения из рейсов, если они есть.

Факт по-прежнему берется из завершенных рейсов.

## Что изменилось для Водителя

Статусбар выполнения сменного плана Водителя должен использовать серверный план техники, если такой план задан.

Если плана техники еще нет, допускается временный fallback на старый черновой расчет, чтобы экран не ломался.

## Ограничение текущего этапа

Ручной план уже является источником для серверного расчета, но не все рабочие экраны обязаны сразу отображать его в полной форме.

Следующий этап - подключить прогресс техники к карточкам самосвалов и экскаваторов у:

- Горного мастера;
- Горного диспетчера;
- Машиниста экскаватора;
- Водителя самосвала.

## Правило для следующих чатов

Нельзя придумывать локальные планы и проценты внутри отдельных интерфейсов.

Если экрану нужен план или процент выполнения, он должен получать эти данные из серверного контура сменных планов и прогресса техники.

# Изоляция ролевых PWA и пользовательских сессий

Дата решения: 16.07.2026.

Статус: реализовано и проверено локально в ветке `codex/role-app-session-isolation`. Commit, push и deploy не выполнялись. Production и `driverform.ru` не изменялись.

## Проблема

Разные PWA, установленные с одного origin `driverform.ru`, используют одну cookie-сессию браузера. Имя, иконка, manifest, `start_url` и scope PWA не создают отдельное хранилище cookie. Поэтому вход Водителем в одном установленном приложении заменял вход Машиниста экскаватора, Горного мастера или другой роли во всех приложениях и вкладках того же origin.

## Принятое решение

Каждое рабочее приложение получает отдельный origin на ролевом поддомене, но продолжает обслуживаться тем же Django-приложением и той же учетной системой:

| Роль | Рабочий origin | Стартовый маршрут |
|---|---|---|
| Водитель самосвала | `driver.driverform.ru` | `/driver/` |
| Машинист экскаватора | `excavator.driverform.ru` | `/excavator/work/` |
| Горный мастер | `mining-master.driverform.ru` | `/mining-master/assignments/` |
| Заместитель начальника горного участка | `deputy.driverform.ru` | `/deputy-mining-manager/` |
| Горный диспетчер | `dispatcher.driverform.ru` | `/dispatcher/control/` |
| ОУП | `oup.driverform.ru` | `/oup/employees/` |
| Механическая служба | `mechanic.driverform.ru` | `/mechanic/downtimes/` |
| Руководство | `management.driverform.ru` | `/reports/management/` |
| Системный администратор | `admin.driverform.ru` | `/system-admin/` |

Корневой `driverform.ru` остается единым универсальным входом для обычной работы и обратной совместимости. Изоляция нескольких одновременных ролей обеспечивается только при открытии соответствующих ролевых origin.

## Контракт сессий

- Cookie сессии и CSRF остаются host-only: `SESSION_COOKIE_DOMAIN = None` и `CSRF_COOKIE_DOMAIN = None`.
- Нельзя задавать общий домен cookie `.driverform.ru`, иначе все ролевые приложения снова начнут делить одну сессию.
- На ролевом origin сервер разрешает вход только для закрепленной за ним роли.
- Чужая или устаревшая сессия на ролевом origin не принимается и очищается перед новым входом.
- При входе ключ сессии ротируется.
- Режим `Личное` получает скользящий срок сессии 365 дней: срок продлевается при использовании приложения.
- Режим `Общее` сохраняет прежний безопасный контракт: сессия закрывается вместе с браузером.
- Вкладки с разными ролевыми поддоменами имеют независимые сессии. Несколько вкладок одного поддомена закономерно используют одну и ту же роль.

## Контракт PWA

- Для всех девяти ролей определены собственные manifest, название, короткое название, цвет темы, `start_url`, service worker и версия shell-кэша.
- На своем ролевом origin приложение получает scope `/`, поэтому экран входа и рабочий маршрут находятся внутри одной установленной PWA.
- На общем origin сохраняется прежний ограниченный scope конкретного рабочего контура.
- Service worker каждой роли удаляет только старые кэши своей семьи и не затрагивает кэши других приложений.
- Произвольный navigation response больше не записывается поверх канонического offline-shell.
- Универсальный экран входа на корневом origin не регистрирует ролевой PWA.

## Иконки приложений

Создано единое семейство Copper Resources из девяти самостоятельных иконок. Общая система: полноцветное поле роли, графитовый технологический угол, медные идентификационные штрихи и крупный бело-медный предметный знак без текста, градиентов и мелких деталей.

| Роль | Основной цвет | Символ |
|---|---|---|
| Водитель | бирюзовый | карьерный самосвал |
| Машинист экскаватора | желтый | экскаватор |
| Горный мастер | синий | каска и контроль |
| Заместитель | зеленый | сетка расстановки |
| Диспетчер | красный | сеть управления |
| ОУП | малиновый | сотрудники и пропуск |
| Механик | оранжевый | шестерня и инструмент |
| Руководство | индиго | производственная динамика |
| Администратор | серо-графитовый | щит и доступ |

Для каждой роли подготовлены PNG `180x180`, `192x192`, `512x512` и отдельный `maskable-512`. Иконки различимы в фактическом размере ярлыка телефона и остаются частью одной визуальной системы.

## Отдельный этап публикации

До production-проверки потребуется отдельное прямое разрешение на инфраструктурный этап:

1. Настроить DNS для девяти поддоменов, при необходимости wildcard `*.driverform.ru`.
2. Подключить TLS-сертификат, покрывающий все ролевые поддомены.
3. Направить все ролевые host в тот же Django upstream через nginx.
4. Добавить `.driverform.ru` или полный список host в production `DJANGO_ALLOWED_HOSTS`.
5. Не задавать `SESSION_COOKIE_DOMAIN` и `CSRF_COOKIE_DOMAIN` родительским доменом.
6. Выполнить отдельный rollback-ready deploy только после разрешения пользователя.
7. Проверить вход во все роли параллельно в разных вкладках и на двух-трех установленных PWA.
8. Удалить старые одно-origin ярлыки с тестового телефона и установить приложения заново с ролевых адресов, чтобы ОС создала независимые приложения с новыми иконками.

Миграция схемы БД для решения не требуется.

# Этапы работы над проектом

Дата создания: 17.06.2026

## Общий принцип

Проект ведем в одной рабочей папке и одной общей памяти проекта.

Не создаем отдельный проект под каждый этап, потому что все этапы связаны одним ядром данных.

Внутри общей папки `ПРОГРЕСС_ПРОЕКТА` создаем отдельные документы по этапам и модулям.

## Почему лучше один общий проект

Причины:

- единая архитектура;
- единая база данных;
- общие справочники;
- общие роли и доступы;
- отчеты зависят от данных разных модулей;
- проще сохранять контекст и решения;
- меньше риска, что разные этапы начнут противоречить друг другу.

## Когда можно делать отдельный проект/ветку

Отдельный проект или отдельную рабочую ветку имеет смысл делать только для:

- разработки кода MVP;
- эксперимента с интерфейсом;
- тестового прототипа;
- отдельной презентации;
- отдельной демонстрационной версии.

Но проектная документация и решения должны оставаться в общей папке прогресса.

## Предлагаемые этапы

### Этап 1. ТЗ MVP производственного контура

Цель: собрать рабочее техническое задание на первый контур.

Состав:

- роли;
- сценарии смены;
- интерфейс водителя;
- интерфейс экскаваторщика;
- интерфейс горного мастера;
- интерфейс диспетчера;
- механика рейсов;
- переходящие рейсы;
- справочники;
- отчеты;
- права доступа;
- аварийные сценарии.

Результат: документ `ТЗ_MVP_ПРОИЗВОДСТВЕННОГО_КОНТУРА.md`.

### Этап 2. Модель данных

Цель: описать структуру базы данных.

Состав:

- сотрудники;
- пользователи;
- роли;
- техника;
- смены;
- назначения;
- рейсы;
- статусы;
- простои;
- ремонты;
- справочники;
- отчеты;
- журнал действий.

Результат: схема данных и описание таблиц на русском языке.

### Этап 3. Прототип интерфейсов

Цель: нарисовать и согласовать первые экраны.

Состав:

- экран водителя;
- экран экскаваторщика;
- экран горного мастера;
- экран диспетчера;
- мобильная PWA-логика;
- диспетчерский экран под компьютер.

Результат: макеты экранов и сценарии пользователя.

### Этап 4. Технический стек и каркас системы

Цель: выбрать и собрать основу приложения.

Предварительное направление:

- Python;
- Django;
- PostgreSQL;
- веб-интерфейс;
- PWA для полевых ролей;
- админка;
- экспорт отчетов.

Результат: рабочий технический каркас.

### Этап 5. MVP рейсов и смен

Цель: реализовать самую важную производственную механику.

Состав:

- открытие смены;
- расстановка;
- создание рейса экскаваторщиком;
- подтверждение назначения водителем;
- закрытие рейса водителем;
- переходящий груженый рейс;
- закрытие смен;
- диспетчерский контроль.

Результат: первый рабочий контур.

### Этап 6. Отчеты первой очереди

Цель: автоматизировать отчеты, которые сейчас перегружают диспетчерскую.

Состав:

- сменный отчет;
- суточный отчет;
- объемы по экскаваторам;
- объемы по самосвалам;
- простои;
- КИП и КТГ;
- отчет заказчику;
- экспорт в Excel/PDF.

Результат: автоматическая генерация базовых отчетов.

### Этап 7. Пилот в реальной смене

Цель: проверить систему на реальном процессе.

Состав:

- тест на одной смене;
- сравнение с текущими Excel-отчетами;
- сбор ошибок;
- корректировка сценариев;
- доработка интерфейсов.

Результат: подтверждение, что MVP можно запускать шире.

## С чего начинаем

Начинаем с этапа 1: `ТЗ MVP производственного контура`.

Для этого уже есть основа:

- 20 ответов по производственному контуру;
- документ по интерфейсам;
- документ по справочникам;
- журнал решений;
- план производственного контура.

Следующий рабочий шаг: собрать единый документ ТЗ.

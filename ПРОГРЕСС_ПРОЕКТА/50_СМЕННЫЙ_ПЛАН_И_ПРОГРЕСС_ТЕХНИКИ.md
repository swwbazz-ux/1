# Сменный план и прогресс техники

Дата фиксации: 04.07.2026.

## Решение

Прогресс выполнения сменного плана должен считаться на сервере.

Рабочие приложения не должны отправлять на сервер готовый процент статусбара как источник истины. Водитель, экскаваторщик, горный мастер и диспетчер отправляют события работы: рейс создан, самосвал загружен, рейс разгружен, простой открыт, простой закрыт, техника переназначена.

Сервер на основе этих событий считает:

- количество рейсов;
- объем, м3;
- тоннаж;
- план;
- факт;
- процент выполнения;
- активный рейс;
- состояние техники;
- время последнего пересчета.

Интерфейсы только отображают серверный расчет.

## Почему не клиентский процент

Статусбар в PWA Водителя является визуальным элементом. Его нельзя считать учетным источником, потому что:

- телефон может быть offline;
- приложение может иметь старый кэш;
- водитель может закрыть рейс позже;
- диспетчер может служебно закрыть или исправить рейс;
- один и тот же процент должны видеть Водитель, Экскаваторщик, Горный мастер и Диспетчер;
- отчеты должны опираться на серверные данные, а не на состояние экрана.

## Масштаб

Фундамент должен сразу выдерживать парк больше 50 самосвалов и больше 10 экскаваторов.

Нельзя строить прогресс как локальный расчет внутри одной страницы. Нужен единый серверный расчет по смене, технике и назначению.

## Текущий статус MVP

В MVP уже есть учетная основа:

- `Trip` хранит рейсы;
- у рейса есть самосвал, экскаватор, водитель, экскаваторщик, смена загрузки, смена разгрузки;
- у рейса есть `volume_m3`, `tonnage`, `planned_volume_m3`;
- отчеты уже агрегируют объемы и тоннаж из рейсов;
- диспетчерские и отчетные экраны уже частично считают суммы по `Trip`.

Но полноценного сменного плана пока нет:

- у Водителя процент сейчас считается черновиком от количества рейсов и условного плана `20`;
- у Диспетчера есть общий план в коде, а не нормальная серверная настройка смены;
- нет единой сущности сменного задания по самосвалу, экскаватору, комплексу или участку;
- нет общего серверного снапшота прогресса техники для передачи всем рабочим экранам.

Вывод: данные рейсов уже собираются, но контур сменного планирования и оперативного прогресса нужно выделить отдельным серверным слоем.

## Целевая модель

Минимальный серверный фундамент:

### `ShiftPlan`

Общий план объема.

Поля:

- тип плана: месячный, суточный, дневная смена или ночная смена;
- техническая дата начала действия для совместимости отчетов;
- расчетная смена для сменных планов;
- план объема, м3;
- активность;
- кто создал;
- комментарий.

Общий `ShiftPlan` не должен содержать рабочий план рейсов или план тоннажа в пользовательской админке.
Рейсы задаются на уровне конкретной техники, тоннаж остается фактическим отчетным показателем.

### `EquipmentShiftPlan`

План по конкретной технике.

Поля:

- смена;
- техника;
- вид техники;
- сотрудник, если план персональный;
- план рейсов для самосвалов;
- план м3 для экскаваторов;
- правило расчета процента: по рейсам или по м3;
- активность.

### `EquipmentShiftProgress`

Расчетный серверный прогресс по технике.

Поля:

- смена;
- техника;
- сотрудник;
- выполнено рейсов;
- выполнено м3;
- выполнено тонн;
- процент выполнения;
- активный рейс;
- состояние техники;
- время последнего пересчета.

На первом этапе `EquipmentShiftProgress` можно не хранить таблицей, а считать функцией `calculate_equipment_shift_progress(...)`. Если нагрузка или интерфейсы начнут часто запрашивать прогресс для 50+ самосвалов, расчет нужно кешировать или сохранять снапшотом.

## Обновление после рейса

После каждого события рейса сервер должен:

1. сохранить изменение рейса;
2. пересчитать прогресс самосвала;
3. пересчитать прогресс экскаватора или комплекса, если это нужно экрану;
4. поднять realtime-событие `trip_changed` или отдельное событие `shift_progress_changed`;
5. отдать в payload идентификаторы затронутой техники и новый процент прогресса.

## Отображение на экранах

Один и тот же серверный процент должен отображаться:

- у Водителя в основном статусбаре;
- у Горного мастера в карточке самосвала;
- у Диспетчера в карточке самосвала;
- у Экскаваторщика в карточке назначенного самосвала.

Экран не должен сам решать, сколько процентов выполнено. Он получает `progress_percent` и рисует статусбар.

## Правило для следующих чатов

Если чат работает с Водителем, Экскаваторщиком, Горным мастером или Диспетчером, он не должен вводить локальный расчет процента выполнения сменного плана.

Нужно подключаться к серверному расчету прогресса техники. Если серверный расчет еще не реализован, сначала надо сделать общий серверный контур, а уже потом рисовать статусбары в иконках техники.

## Обновление 04.07.2026: первый ручной контур реализован

Добавлены серверные сущности:

- `ShiftPlan` - общий план объема с типом `месячный`, `суточный`, `дневная смена` или `ночная смена`;
- `EquipmentShiftPlan` - план конкретной техники внутри сменного плана.

В админке MVP добавлены разделы:

- `Сменные планы`;
- `Планы техники`.

Серверный сервис расчета прогресса техники считает факт по завершенным рейсам:

- для самосвала - по смене разгрузки;
- для экскаватора - по смене загрузки.

Витрина руководства использует ручные сменные планы как источник планового объема. В сменный расчет попадают только планы типов `дневная смена` и `ночная смена`; суточный и месячный планы не должны суммироваться со сменными, чтобы не было двойного счета. Если ручного плана нет, остается резервный расчет по плановым значениям рейсов.

Статусбар Водителя использует серверный план техники, если он задан. Если плана нет, временно сохраняется fallback на старый черновой расчет.

Следующий шаг: подключить этот серверный прогресс к карточкам самосвалов и экскаваторов у Горного мастера, Горного диспетчера и Машиниста экскаватора.

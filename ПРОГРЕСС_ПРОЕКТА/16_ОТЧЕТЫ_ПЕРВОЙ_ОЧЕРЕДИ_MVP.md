# Отчеты первой очереди MVP

Дата создания: 17.06.2026

## Назначение документа

Этот документ фиксирует первые отчеты, которые должны появиться в MVP производственного контура.

Главный приоритет - автоматизировать учет объемов за смену и за сутки, потому что именно эта часть сейчас создает основную нагрузку на диспетчерскую службу.

Внешний обязательный срок: суточный отчет заказчику должен быть готов к 08:00.

## Главная цель отчетного блока

Система должна собирать данные из карьера в течение смены так, чтобы диспетчерская не пересобирала объемы вручную из множества таблиц.

Отчеты должны формироваться из событий, которые уже произошли в системе:

- назначение самосвала под экскаватор;
- создание рейса машинистом экскаватора;
- завершение рейса водителем;
- выбранная порода;
- выбранная точка разгрузки;
- смена;
- техника;
- сотрудник;
- объем по справочнику техники и породы.

## Первый отчетный приоритет

### 1. Объемы за смену

Этот отчет нужен для понимания результата дневной или ночной смены.

Минимальный состав:

- дата;
- смена: дневная или ночная;
- экскаватор;
- машинист экскаватора;
- самосвал;
- водитель;
- порода;
- точка разгрузки;
- количество рейсов;
- объем;
- тоннаж, если нужен расчет по плотности;
- переходящие рейсы, если самосвал ушел на пересменку груженым;
- примечания или признаки корректировки.

Основная задача: показать, какие объемы были выработаны за конкретную смену и кем.

### 2. Объемы за сутки

Суточный отчет собирает дневную и ночную смену в один период.

Минимальный состав:

- дата производственных суток;
- дневная смена;
- ночная смена;
- общий объем за сутки;
- объем по экскаваторам;
- объем по самосвалам;
- объем по породам;
- объем по точкам разгрузки;
- количество рейсов;
- расхождения или незакрытые события;
- переходящие рейсы между сменами.

Основная задача: дать сводную картину по суткам без ручного пересчета.

### 3. Суточный отчет заказчику к 08:00

Этот отчет должен быть готов к 08:00.

Он должен брать данные из суточного отчета, но показывать их в форме, подходящей для передачи заказчику.

Минимальный состав пока фиксируется как черновой:

- дата отчетных суток;
- общий объем;
- объем по видам породы;
- объем по точкам разгрузки или направлениям;
- объем по экскаваторам, если это требуется заказчику;
- объем по самосвалам, если это требуется заказчику;
- итоговые значения;
- подпись/ответственный или служебная отметка формирования отчета.

Точная форма заказчика должна быть сверена с текущим Excel-отчетом.

## Как система должна считать объем

Базовая логика MVP:

1. Машинист экскаватора создает рейс: самосвал + точка разгрузки + текущая порода/параметры экскаватора.
2. Система фиксирует смену загрузки.
3. Водитель завершает рейс кнопкой `Выполнено`.
4. Система фиксирует смену разгрузки.
5. Для объема экскаватора используется смена загрузки.
6. Для работы водителя используется смена разгрузки.
7. Если рейс переходит через пересменку, система сохраняет это как переходящий рейс.

Предварительный расчет объема:

- объем рейса берется из справочника техники и породы;
- для самосвала учитывается модель, грузоподъемность и расчетная кубатура;
- для породы учитывается справочник пород и коэффициентов;
- тоннаж может рассчитываться через плотность, если это требуется отчетом.

## Какие данные должны быть в ядре для отчетов

Для отчетов первой очереди в базе должны быть:

- дата и время создания рейса;
- дата и время завершения рейса;
- отчетная смена загрузки;
- отчетная смена разгрузки;
- экскаватор;
- машинист экскаватора;
- самосвал;
- водитель;
- порода;
- точка разгрузки;
- объем рейса;
- тоннаж рейса, если требуется;
- признак переходящего рейса;
- статус рейса;
- кто создал рейс;
- кто завершил рейс;
- кто исправлял рейс, если была корректировка.

## Источники данных

### Машинист экскаватора

Дает системе:

- текущий экскаватор;
- горизонт;
- блок;
- породу;
- точки разгрузки;
- создание рейса;
- простои и статусы экскаватора.

### Водитель самосвала

Дает системе:

- подтверждение начала смены;
- топливо, пробег, моточасы на начало смены;
- подтверждение нового назначения;
- завершение рейса кнопкой `Выполнено`;
- закрытие смены.

### Горный мастер

Дает системе:

- расстановку самосвалов под экскаваторы;
- изменения назначений;
- контроль работы комплексов;
- при необходимости переназначение техники.

### Диспетчер

Дает системе:

- контроль смены;
- служебное закрытие смены при необходимости;
- корректировки в рамках прав;
- формирование и проверку отчетов.

### Администратор

Дает системе:

- справочники техники;
- справочники пород;
- точки разгрузки;
- роли и доступы;
- правила и настройки отчетного периода.

## Важные правила отчетности

### Правило 1. Отчетных смен только две

Несмотря на раннюю пересменку части комплексов, отчетных смен всего две:

- дневная;
- ночная.

Ранняя пересменка не создает отдельный отчетный период.

### Правило 2. Сутки для отчета заказчику

Суточный отчет заказчику должен быть готов к 08:00.

Это значит, что система должна заранее показывать диспетчеру:

- какие рейсы не закрыты;
- какие смены не закрыты;
- где есть расхождения;
- какие данные требуют проверки до формирования отчета.

### Правило 3. Переходящий груженый рейс

Если самосвал ушел на пересменку груженым:

- рейс сохраняется активным;
- следующий водитель видит активный рейс;
- объем экскаватора относится к смене загрузки;
- работа водителя относится к смене разгрузки;
- в отчете должен быть признак переходящего рейса.

### Правило 4. Исправления должны быть видны

Если диспетчер или другой ответственный сотрудник исправляет данные рейса или отчета, система должна сохранить:

- кто исправил;
- когда исправил;
- что было изменено;
- причину, если она требуется для конкретного вида исправления.

## Первая отчетная витрина диспетчера

До формирования итогового отчета диспетчер должен видеть рабочую витрину смены.

Минимальный состав витрины:

- текущая дата;
- текущая смена;
- активные экскаваторы;
- активные самосвалы;
- рейсы в работе;
- закрытые рейсы;
- переходящие рейсы;
- объем за текущую смену;
- объем за текущие сутки;
- незакрытые смены сотрудников;
- незакрытые рейсы;
- предупреждения по данным.

Цель витрины: не ждать конца смены, а видеть проблему до того, как отчет понадобится к 08:00.

## Первая версия отчета для руководства

Для руководства на первом этапе нужна не детальная таблица, а краткая витрина:

- объем за текущую смену;
- объем за текущие сутки;
- план/факт, если план будет задан;
- объем по экскаваторам;
- объем по направлениям;
- количество рейсов;
- техника в работе;
- техника в простое;
- главные отклонения.

Эта витрина не является главным MVP, но ее можно сделать на основе тех же данных.

## Конструктор отчетов

Пользователь предложил сделать конструктор отчетов, где администратор или ответственный сотрудник сможет сам выбирать, какие данные отображать в столбцах, сохранять собранный отчет под своим названием и выгружать его в Excel.

Идея правильная для будущей системы, потому что отчеты будут меняться: для диспетчерской, руководства, заказчика, механиков и других отделов могут понадобиться разные представления одних и тех же данных.

Но для первого MVP нужно разделить это на два уровня.

### Уровень 1. Конструктор отчетов MVP

В MVP делаем упрощенный конструктор только для отчетов по объемам.

Он должен позволять:

- выбрать тип отчета: сменный, суточный, отчет заказчику;
- выбрать период;
- выбрать смену, если отчет сменный;
- выбрать доступные столбцы из разрешенного списка;
- изменить порядок столбцов;
- сохранить шаблон отчета под своим названием;
- открыть сохраненный шаблон;
- выгрузить отчет в Excel.

Права в MVP:

- создавать и изменять шаблоны отчетов может только администратор;
- диспетчер использует готовые шаблоны и выгружает отчеты;
- в будущем право создания и изменения шаблонов можно дать диспетчерам.

Примеры доступных столбцов для MVP:

- дата;
- смена;
- экскаватор;
- машинист экскаватора;
- самосвал;
- водитель;
- порода;
- точка разгрузки;
- количество рейсов;
- объем;
- тоннаж;
- переходящий рейс;
- примечание;
- признак корректировки.

Такой вариант дает гибкость, но не ломает сроки первого запуска.

### Уровень 2. Полный конструктор отчетов

Полный конструктор отчетов - отдельный большой модуль после запуска MVP.

Он может позволять:

- собирать отчеты по любым модулям системы;
- выбирать источники данных;
- задавать группировки;
- задавать фильтры;
- добавлять вычисляемые поля;
- строить графики;
- сохранять разные формы отчетов для разных ролей;
- настраивать доступ к отчетам;
- делать автоматическую рассылку или выгрузку по расписанию.

Такой модуль полезен, но если делать его сразу полностью, он может сильно задержать запуск производственного учета.

### Рабочее решение

Для MVP фиксируем компромисс:

- не делаем сразу сложный универсальный конструктор всех отчетов;
- делаем простой конструктор шаблонов для отчетов по объемам;
- обязательно добавляем выгрузку в Excel;
- архитектуру отчетов проектируем так, чтобы позже расширить ее до полноценного конструктора.

## Что нужно найти в текущих Excel-файлах

Для точного повторения отчетов нужно разобрать текущие Excel-формы:

- сменный отчет диспетчерской;
- суточный отчет диспетчерской;
- отчет заказчику;
- таблицы объемов по самосвалам;
- таблицы объемов по экскаваторам;
- отчеты по направлениям/точкам разгрузки;
- формулы расчета объема и тоннажа;
- правила округления;
- обязательные поля для заказчика.

## Критерии готовности отчетного блока MVP

Отчетный блок первой очереди можно считать готовым, если:

- система формирует объемы за смену;
- система формирует объемы за сутки;
- суточный отчет заказчику можно получить к 08:00;
- данные строятся из рейсов, а не вводятся вручную повторно;
- видны незакрытые рейсы и смены;
- переходящие рейсы учитываются без ручного пересчета;
- диспетчер может проверить данные перед выгрузкой;
- отчет можно выгрузить в Excel;
- можно сохранить хотя бы один пользовательский шаблон отчета по объемам с выбранными столбцами;
- исправления фиксируются в журнале.

## Текущий вывод

Первый отчетный блок MVP должен быть построен вокруг одной цели: к 08:00 заказчик и компания должны получить понятные суточные объемы без ручной сборки данных диспетчерами.

Для этого сначала нужно сделать не красивый отчет, а правильную цепочку данных:

рейс -> смена -> сутки -> объем -> проверка диспетчером -> выгрузка.

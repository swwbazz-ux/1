# Модель данных MVP производственного контура

Дата создания: 17.06.2026

## Назначение документа

Этот документ описывает первую модель данных для MVP производственного контура.

Модель данных нужна для того, чтобы заранее понять:

- какие данные будут храниться в ядре системы;
- какие данные являются справочниками;
- какие данные появляются каждый день в рабочей смене;
- какие связи нужны между сотрудниками, техникой, сменами, рейсами и отчетами;
- какие данные должны попадать в отчеты диспетчерской службы.

Это не финальная структура базы данных для программиста, а рабочая проектная схема на русском языке. Позже на ее основе можно будет сделать техническую схему базы данных.

## Главный принцип

В системе есть два типа данных:

1. **Справочники**  
   Это относительно постоянные данные: сотрудники, техника, породы, точки разгрузки, причины простоев, участки, горизонты, блоки, смены, роли.

2. **События смены**  
   Это то, что происходит в работе каждый день: открытие смены, назначение техники, рейсы, простои, перегон экскаватора, изменение точки разгрузки, закрытие смены.

Справочники создают основу, а события смены строятся поверх этой основы.

## Основные сущности ядра системы

### 1. Сотрудники

Содержит всех людей, которые участвуют в системе.

Примеры ролей:

- водитель самосвала;
- экскаваторщик;
- горный мастер;
- диспетчер;
- механик;
- администратор;
- руководитель.

Основные данные:

- ФИО;
- табельный номер;
- должность;
- отдел;
- статус сотрудника: работает, в отпуске, уволен, временно неактивен;
- роль в системе;
- доступ к рабочему кабинету;
- закрепленная техника, если применимо;
- дата приема;
- дата увольнения, если применимо.

Важно: сотрудник может быть в общей кадровой системе, но в MVP производственного контура нам нужны только те данные, которые необходимы для смен, назначений, рейсов и отчетов.

### 2. Роли и доступы

Определяет, какой интерфейс открывается сотруднику и какие действия он может выполнять.

Примеры ролей:

- администратор;
- диспетчер;
- горный мастер;
- водитель самосвала;
- экскаваторщик;
- механик;
- руководство.

Основные данные:

- название роли;
- доступные экраны;
- разрешенные действия;
- право создавать;
- право редактировать;
- право закрывать смены других сотрудников;
- право смотреть отчеты.

Отдельное правило: диспетчер должен иметь право закрыть смену водителя или экскаваторщика в аварийной ситуации.

Роль также должна определять, с каким видом техники может работать сотрудник.

Примеры:

- водитель самосвала -> самосвал;
- экскаваторщик -> экскаватор;
- механик -> техника по ремонтным заявкам;
- горный мастер -> расстановка техники;
- диспетчер -> контроль техники и смен.

### 2.1. Учетные записи и первичный доступ

Фиксирует вход пользователя в систему.

Основные данные:

- сотрудник;
- логин или номер телефона, если используется;
- пинкод или пароль;
- роль;
- интерфейс, который открывается после входа;
- статус доступа: создан, активен, заблокирован, удален;
- дата создания доступа;
- кто создал доступ;
- дата первого входа;
- признак прохождения первичной регистрации.

Администратор создает доступ, а сотрудник при первом входе заполняет данные, необходимые для начала работы.

Это снижает ручную нагрузку на администратора: водитель сам указывает смену, технику и место проживания, а ядро проверяет ограничения.

Правило: пинкод или пароль должен быть привязан к конкретному сотруднику и роли. После входа система открывает интерфейс по роли: водитель, экскаваторщик, диспетчер, горный мастер, механик или администратор.

Если у сотрудника несколько ролей, нужно отдельное правило: либо одна учетная запись с выбором роли после входа, либо отдельные рабочие доступы по ролям.

### 2.2. Первичная регистрация водителя

Фиксирует первичную привязку водителя к рабочим параметрам.

Основные данные:

- водитель;
- выбранная смена: день или ночь;
- выбранный самосвал;
- общежитие;
- блок общежития;
- секция;
- дата регистрации;
- статус регистрации: ожидает, завершена, отклонена, требует проверки;
- причина отказа, если система не разрешила регистрацию.

После успешной регистрации система может допустить водителя к открытию смены.

После первичной регистрации водитель уже не меняет технику самостоятельно. Дальнейшие изменения выполняет диспетчер, горный мастер или администратор, а водитель только подтверждает новое назначение.

### 2.3. Разрешенные виды техники по роли

Определяет, какие виды техники доступны сотруднику в зависимости от роли.

Основные данные:

- роль;
- вид техники;
- можно выбирать самостоятельно при регистрации;
- можно назначать только администратором;
- можно назначать горным мастером;
- можно назначать диспетчером;
- активность правила.

Это нужно, чтобы система не была зашита только под самосвалы.

### 3. Техника

Содержит всю технику, которая участвует в производственном процессе.

Типы техники:

- самосвал;
- экскаватор;
- вспомогательная техника;
- подрядная техника;
- временная техника.

Основные данные:

- гаражный номер;
- тип техники;
- модель;
- грузоподъемность;
- объем кузова или ковша;
- владелец: своя, подрядная;
- статус: активна, временная, выведена, в ремонте, списана;
- дата начала использования;
- дата окончания использования, если техника временная;
- примечание.

Для самосвалов дополнительно:

- грузоподъемность;
- объем кузова;
- расход топлива, если нужен для аналитики;
- текущий статус: свободен, назначен, в рейсе, груженый, на разгрузке, простой, ремонт, ТО.

Для экскаваторов дополнительно:

- объем ковша;
- производительность, если нужна для аналитики;
- текущий статус: в работе, простой, ремонт, перегон, ожидание, неактивен.
- для текущего производственного контура нужно учитывать 8 собственных экскаваторов компании.

### 4. Смены

Содержит производственные смены.

Основные данные:

- дата смены;
- тип смены: дневная, ночная;
- время начала;
- время окончания;
- статус смены: планируется, открыта, закрыта, проверяется, утверждена;
- ответственный диспетчер;
- горный мастер;
- комментарий.

Смена является главным контейнером для всех событий: назначений, рейсов, простоев и показателей техники.

### 4.1. Вахты

Фиксирует период, в рамках которого действует ограничение по составу водителей на один самосвал и по проживанию.

Основные данные:

- дата начала вахты;
- дата окончания вахты;
- статус вахты: планируется, активна, закрыта;
- комментарий.

Правило: в одну вахту на один самосвал может быть привязано не более 2 водителей: один на дневную смену и один на ночную.

Точная длительность и правила определения вахты требуют уточнения.

### 5. Открытие смены сотрудником

Фиксирует, что конкретный сотрудник начал работу в конкретной смене.

Основные данные:

- сотрудник;
- роль на смене;
- смена;
- техника, если сотрудник работает на технике;
- время открытия смены;
- стартовые показатели техники;
- статус открытия: открыта, закрыта, закрыта диспетчером.

Для водителя самосвала стартовые показатели:

- топливо на начало смены;
- моточасы на начало смены;
- пробег на начало смены;
- состояние самосвала на начало смены;
- переходящий груженый рейс, если он есть.

Для экскаваторщика:

- экскаватор;
- начальный горизонт;
- начальный блок;
- начальная порода;
- начальная точка или точки разгрузки;
- начальный статус экскаватора.

### 6. Закрытие смены сотрудником

Фиксирует завершение работы сотрудника в смене.

Основные данные:

- сотрудник;
- смена;
- время закрытия;
- кто закрыл: сам сотрудник или диспетчер;
- причина закрытия диспетчером, если применимо;
- итоговые показатели техники;
- наличие незавершенных рейсов;
- наличие переходящего груженого рейса;
- комментарий.

Для водителя:

- топливо на конец смены;
- моточасы на конец смены;
- пробег на конец смены;
- статус самосвала на конец смены;
- груженый или пустой самосвал на момент закрытия.

Для экскаваторщика:

- итоговый статус экскаватора;
- список простоев;
- список перегонов;
- список отправленных рейсов;
- текущий горизонт, блок, порода и точки разгрузки на момент закрытия.

### 7. Назначения техники

Фиксирует, кто на какой технике работает и под каким экскаватором находится самосвал.

Основные данные:

- смена;
- горный мастер или диспетчер, который сделал назначение;
- водитель;
- самосвал;
- экскаватор;
- время назначения;
- время подтверждения водителем;
- статус подтверждения: ожидает, принято;
- причина переназначения, если оно было;
- активность назначения.

Важное правило: водитель не выбирает себе экскаватор сам. Назначение делает горный мастер или диспетчер.

### 7.1. Привязка сотрудника к технике на вахту

Фиксирует, какие сотрудники закреплены за конкретной техникой в рамках вахты.

Основные данные:

- вахта;
- сотрудник;
- роль;
- вид техники;
- конкретная техника;
- смена: день или ночь;
- статус привязки: активна, отменена, завершена;
- дата создания;
- кто создал или подтвердил;
- комментарий.

Ограничения:

- один самосвал не может иметь двух активных водителей в одну и ту же смену;
- в одну вахту на самосвал допускается до 2 активных водителей: день и ночь;
- один экскаватор не должен иметь двух активных экскаваторщиков в одну и ту же смену, если отдельным правилом не разрешено иначе;
- всего на один самосвал может быть до 4 водителей в более широком периоде, но это правило нужно уточнить отдельно.

### 7.2. Пересадка сотрудника на другую технику

Фиксирует смену техники сотрудника в течение смены или вахты.

Основные данные:

- сотрудник;
- роль;
- вид техники;
- старая техника;
- новая техника;
- смена;
- вахта;
- время запроса;
- время подтверждения;
- инициатор: сотрудник, диспетчер, горный мастер, администратор;
- кто подтвердил;
- причина пересадки;
- статус: ожидает, подтверждена, отклонена, отменена;
- комментарий.

Правило MVP:

- после первичной регистрации сотрудник не меняет технику самостоятельно;
- диспетчер или горный мастер назначает новую технику;
- сотрудник подтверждает новое назначение кнопкой `Принял`;
- система проверяет, что новая техника не занята в текущей смене другим сотрудником с конфликтующей ролью;
- все пересадки и подтверждения фиксируются в журнале действий.

### 8. Работа экскаватора

Фиксирует состояние экскаватора внутри смены.

Основные данные:

- смена;
- экскаватор;
- экскаваторщик;
- текущий горизонт;
- текущий блок;
- текущая порода;
- доступные точки разгрузки;
- текущий статус;
- время изменения статуса;
- активные самосвалы под экскаватором.

Экскаваторщик сам меняет рабочие параметры экскаватора, потому что экскаватор может несколько раз за смену менять горизонт, блок, породу и место работы.

### 9. События статуса экскаватора

Фиксирует простои, перегоны и другие состояния экскаватора.

Основные данные:

- экскаватор;
- экскаваторщик;
- смена;
- тип события: работа, простой, перегон, ремонт, ожидание;
- причина из справочника;
- время начала;
- время окончания;
- длительность;
- комментарий;
- признак критичности;
- нужно ли уведомить диспетчера;
- нужно ли уведомить механиков;
- кто завершил событие.

Время начала и окончания должно фиксироваться автоматически по факту изменения статуса.

### 10. Рейсы

Фиксирует каждую перевозку породы самосвалом.

Основные данные:

- смена погрузки;
- смена разгрузки;
- экскаватор;
- экскаваторщик;
- самосвал;
- водитель;
- точка погрузки;
- точка разгрузки;
- порода;
- горизонт;
- блок;
- время создания рейса;
- время разгрузки;
- статус рейса;
- объем;
- тоннаж, если нужен;
- признак переходящего рейса;
- кто создал рейс;
- кто завершил рейс.

Статусы рейса:

- создан;
- в пути;
- груженый на пересменке;
- разгружен;
- отменен;
- исправлен диспетчером.

Важное правило учета:

- для экскаватора объем относится к смене погрузки;
- для водителя самосвала выполненный рейс относится к тому водителю, который довез и нажал `Выполнено`;
- если рейс переходит через пересменку, система должна хранить это как переходящий рейс, а не терять связь между погрузкой и разгрузкой.

### 11. Точки разгрузки

Справочник мест, куда самосвалы могут отвозить породу.

Основные данные:

- название точки;
- тип точки;
- активна или нет;
- заказчик, если применимо;
- какие породы принимаются;
- примечание.

Примеры:

- ККД;
- СКДР;
- отвал;
- склад;
- ОФР;
- временная точка разгрузки.

### 12. Породы

Справочник пород и материалов.

Основные данные:

- название породы;
- группа породы;
- коэффициенты, если нужны;
- плотность, если нужна;
- признак активности;
- примечание.

Этот справочник нужен для отчетов по объемам и для корректной аналитики.

### 13. Горизонты и блоки

Справочники мест ведения работ.

Основные данные горизонта:

- название горизонта;
- карьер или участок;
- активность.

Основные данные блока:

- название блока;
- горизонт;
- активность;
- примечание.

Экскаваторщик должен быстро выбирать актуальный горизонт и блок в своем интерфейсе.

### 13.1. Общежития и проживание

Фиксирует место проживания сотрудника для контроля первичной регистрации.

На MVP это не полный модуль общежитий, а минимальная проверка допустимого заселения.

Справочник общежитий:

- номер общежития;
- активность;
- примечание.

На старте доступны только общежития 5 и 6.

Справочник блоков общежитий:

- общежитие;
- номер блока;
- активность.

Справочник секций:

- общежитие;
- блок;
- секция: А или Б;
- максимальное количество дневных сотрудников;
- максимальное количество ночных сотрудников;
- активность.

Привязка проживания сотрудника:

- сотрудник;
- вахта или период;
- общежитие;
- блок;
- секция;
- смена: день или ночь;
- статус: активна, завершена, отменена.

Ограничение: в одной секции допускается не более 3 человек в день и 3 человек в ночь.

### 14. Простои самосвала

Фиксирует простои самосвала внутри смены.

Основные данные:

- самосвал;
- водитель;
- смена;
- причина простоя;
- время начала;
- время окончания;
- длительность;
- кто зафиксировал;
- комментарий.

На первом этапе можно оставить простои самосвала в минимальном виде, чтобы не перегружать водителя.

### 15. Механические заявки и ремонты

Этот блок нужен для связи с механической службой.

Основные данные:

- техника;
- кто подал сигнал;
- время подачи;
- тип проблемы;
- статус заявки: новая, принята, в работе, завершена;
- механик или ответственный;
- классификация причины механиком;
- время начала ремонта;
- время окончания ремонта;
- комментарий механика.

Правило: экскаваторщик или водитель не должны подробно классифицировать ремонт. Они только сообщают о проблеме или выбирают общий статус. Детализацию делает механическая служба.

### 16. Уведомления

Фиксирует важные события, которые должны увидеть другие участники смены.

Основные данные:

- тип уведомления;
- источник уведомления;
- получатели;
- связанная техника;
- связанный рейс или простой;
- время создания;
- статус: новое, прочитано, принято в работу, закрыто;
- критичность;
- текст уведомления.

Примеры уведомлений:

- экскаватор встал в критический простой;
- самосвал ожидает нового назначения;
- водитель не подтвердил новое назначение;
- рейс стал переходящим на пересменке;
- смена не закрыта вовремя.

### 17. Журнал действий

Фиксирует историю важных действий в системе.

Основные данные:

- кто сделал действие;
- когда сделал;
- что изменил;
- старое значение;
- новое значение;
- связанная смена;
- связанная техника;
- связанный сотрудник;
- причина изменения, если нужна.

Журнал действий нужен для доверия к системе и для разбора спорных ситуаций.

## Справочники первой очереди

Справочники нужно начинать сразу после первичной модели данных, потому что без них невозможно нормально строить смены, рейсы и отчеты.

Первая очередь справочников:

1. Сотрудники.
2. Роли.
3. Техника.
4. Типы техники.
5. Точки разгрузки.
6. Породы.
7. Горизонты.
8. Блоки.
9. Общежития.
10. Блоки общежитий.
11. Секции общежитий.
12. Вахты.
13. Причины простоев экскаватора.
14. Причины простоев самосвала.
15. Статусы техники.
16. Типы смен.
17. Участки или карьеры.
18. Заказчики, если отчеты разделяются по заказчикам.

## Что можно сделать в MVP упрощенно

Чтобы не раздуть первую версию, можно упростить:

- не делать сложную кадровую систему, а хранить только сотрудников, нужных для производственного контура;
- не делать полный модуль общежитий, а оставить только контроль общежития, блока и секции при регистрации водителя;
- не делать глубокий ремонтный учет, а оставить только механические заявки и статусы;
- не делать сложную систему прав, а начать с фиксированных ролей;
- не делать сложную аналитику по топливу, а только сохранять начало и конец смены;
- не делать конструктор отчетов сразу, а сначала воспроизвести обязательные отчеты диспетчерской.

## Что важно не упростить

Нельзя упрощать до потери смысла:

- связь водитель - самосвал - экскаватор - точка разгрузки;
- учет рейса;
- пересменку с груженым самосвалом;
- смены и закрытие смен;
- первичную регистрацию водителя и контроль конфликтов по самосвалу, смене, вахте и месту проживания;
- автоматическую фиксацию времени событий;
- историю изменений;
- отчетность по объемам.

## Открытые вопросы

1. Какие именно отчеты диспетчерской нужно повторить в первой версии.
2. Какие поля обязательны для отчета заказчику.
3. Нужно ли в MVP учитывать заказчика на уровне точки разгрузки, породы или рейса.
4. Как точно считать объем рейса: по грузоподъемности, по объему кузова, по коэффициенту породы или по другой логике.
5. Какие стартовые справочники нужно заполнить до запуска пилота.
6. Кто будет иметь право редактировать каждый справочник.

## Следующий шаг

Следующий документ после модели данных: описание справочников первой очереди.

В нем нужно для каждого справочника определить:

- кто отвечает за заполнение;
- кто может редактировать;
- какие поля обязательны;
- какие значения должны быть в первой версии;
- какие справочники можно пополнять во время смены;
- какие справочники можно менять только через администратора.

## Обновление модели 18.06.2026

После разбора суточного отчета заказчику в сущность `Рейс` добавлены дополнительные производственные поля:

- плановое задание, м3;
- горизонт погрузки;
- блок;
- плечо транспортировки, км;
- простои;
- примечание.

Назначение этих полей: приблизить суточный отчет заказчику к текущей Excel-форме, где по каждой смене отображается работа выемочного оборудования.

Текущее решение MVP: хранить эти параметры на уровне рейса, потому что рейс уже является основной учетной единицей связки экскаватор - самосвал - порода - точка разгрузки.

Будущее уточнение: если ввод этих параметров для каждого рейса окажется лишней нагрузкой для машиниста экскаватора, нужно добавить отдельную сущность текущих параметров экскаватора или смены экскаватора и подставлять эти значения в рейсы автоматически.

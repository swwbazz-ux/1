# Брендбук админки MVP

Дата: 26.06.2026.

Документ нужен как единый ориентир для редизайна экранов админки. Его можно передавать в отдельный чат, который занимается интерфейсами админки, как обязательное ТЗ по визуальному стилю и компонентам.

## Цель

Админка должна выглядеть как часть той же производственной учетной системы, что и пульты Горного диспетчера и Горного мастера, но не должна копировать их боевой диспетчерский экран буквально.

Пульт смены - это плотный операторский экран с максимальной концентрацией информации. Админка - это рабочее место управления сотрудниками, доступами, справочниками, конфликтами, журналами и выгрузками. Поэтому стиль должен быть:

- строгий;
- плотный;
- спокойный;
- технологичный;
- без декоративной маркетинговой подачи;
- с теми же статусными цветами и геометрией, что у пультов.

## Главный Визуальный Принцип

Админка строится как рабочая консоль с двумя режимами смены: `День` и `Ночь`. Дневная тема остается основной для первого визуального прохода, но ночная тема должна быть заложена сразу, чтобы админка не отрывалась от диспетчерского и мастерского пультов.

Переключение `День / Ночь` должно быть оформлено как в диспетчерской: компактная служебная кнопка в правой части шапки рядом с `Назад` и `Выйти`, без отдельной большой панели настроек и без декоративного тумблера в стиле SaaS.

Использовать максимум полезного пространства, но не превращать экран в хаос. Плотность должна быть управляемой: таблицы, списки, фильтры, компактные карточки и боковые панели.

## Шрифты

### Основной шрифт интерфейса

Использовать `Roboto Condensed`, как в пультах диспетчера и мастера.

CSS:

```css
font-family: "Roboto Condensed", "Arial Narrow", "Segoe UI", Arial, sans-serif;
letter-spacing: 0;
```

### Почему не Manrope

В старой админке местами используется `Manrope`. Для нового оформления админки его не использовать как основной, потому что он визуально отрывает админку от пультов. Весь новый административный интерфейс должен перейти на `Roboto Condensed`.

### Масштаб типографики

| Назначение | Размер | Вес | Регистр |
|---|---:|---:|---|
| Заголовок страницы | 32-40 px | 900 | верхний регистр допустим |
| Заголовок панели | 18-22 px | 900 | обычный или верхний |
| Заголовок карточки/строки | 15-17 px | 800-900 | обычный |
| Текст строки таблицы | 13-15 px | 700-800 | обычный |
| Подпись/метаданные | 11-12 px | 700-800 | обычный |
| Кнопки | 12-14 px | 900 | верхний регистр |
| Чипы/бейджи | 10-12 px | 900 | верхний регистр или короткий текст |

Не использовать отрицательный `letter-spacing`. Не масштабировать шрифт от ширины viewport. Для длинных слов использовать перенос или `text-overflow: ellipsis`.

### Переключатель День / Ночь

Переключатель темы админки должен повторять логику диспетчерской:

- находится в правом служебном блоке шапки рядом с `Назад` и `Выйти`;
- высота `38-40 px`;
- ширина `38-44 px`, если используется иконка солнца/луны, или `76-92 px`, если используется текст `День` / `Ночь`;
- радиус `6-7 px`;
- обводка `1px solid var(--admin-line)`;
- фон `var(--admin-panel-soft)`;
- текст/иконка `var(--admin-muted)`;
- активное состояние подсвечивает текущий режим, но не выглядит как основная зеленая кнопка действия;
- hover меняет фон и обводку, без изменения размера.

Рекомендуемый HTML-паттерн:

```html
<button type="button" class="admin-theme-button" data-admin-theme-toggle title="День / Ночь">Д/Н</button>
```

Рекомендуемая JS-логика:

```js
var shell = document.querySelector("[data-admin-theme]");
var saved = localStorage.getItem("admin-theme") || "day";

function setAdminTheme(theme) {
    shell.classList.toggle("admin-night", theme === "night");
    shell.classList.toggle("admin-day", theme !== "night");
    localStorage.setItem("admin-theme", theme);
}
```

Классы темы:

- `admin-day` - дневная тема;
- `admin-night` - ночная тема;
- `data-admin-theme` - корневой контейнер админки, на котором переключаются классы.

Админка не должна использовать `dispatcher-day` / `dispatcher-night` напрямую. Нужно повторить паттерн диспетчерской, но с собственными классами `admin-day` / `admin-night`, чтобы не связать административные экраны с CSS пульта.

## Статусная Семантика

Статусные цвета должны означать одно и то же на всех экранах.

| Статус | Цвет | Где применять |
|---|---|---|
| Норма / активно / работает | зеленый | активный сотрудник, активный доступ, успешная проверка |
| Ожидает / требует внимания | желтый | не активирован, нет данных, ожидание подтверждения |
| Ошибка / конфликт / заблокировано | красный | конфликт, блокировка, опасное действие |
| Информация / переход / служебное | голубой | ссылки, выбранный раздел, технический статус |
| Архив / выключено / нет данных | серый | архив, деактивировано, пустые состояния |

Запрещено использовать один и тот же цвет для разных смыслов на соседних экранах.

## Геометрия

Админка должна использовать небольшие радиусы, как рабочий интерфейс, а не большие мягкие SaaS-карточки.

| Элемент | Радиус |
|---|---:|
| Основная панель | 8 px |
| Табличная строка / список | 6-8 px |
| Поле ввода | 5-6 px |
| Обычная кнопка | 5-6 px |
| Чип / бейдж | 4-6 px |
| Модальное окно | 10-12 px |
| Фото сотрудника | 8 px |

Не использовать `border-radius: 999px` для основных кнопок и навигации. Круглые/пилюльные элементы допустимы только для маленьких индикаторов и коротких бейджей.

## Сетка И Отступы

### Базовый шаг

Основной шаг сетки: `4 px`.

Рабочие отступы:

- `4 px` - микрозазор внутри плотных элементов;
- `6 px` - чипы, мини-кнопки;
- `8 px` - стандартный внутренний зазор;
- `12 px` - зазор между связанными блоками;
- `16 px` - внутренний отступ панели;
- `20-24 px` - крупный отступ страницы или формы.

### Ширина страниц

Для desktop-админки:

```css
width: min(1760px, calc(100vw - 32px));
margin: 0 auto;
```

Для широких таблиц и справочников можно использовать почти всю ширину экрана, но горизонтальный скролл страницы запрещен. Скролл допускается внутри таблицы или списка.

### Высота

Рабочие экраны админки по возможности должны помещаться в один desktop viewport. Если данных много, скроллится рабочая область списка/таблицы, а не вся страница с плавающей шапкой.

## Шапка Админки

Шапка должна быть единой на всех экранах админки.

### Структура

1. Слева: бейдж активного администратора.
2. Центр: название раздела и короткий контекст.
3. Справа: основные переходы и выход.

### Рекомендуемые размеры

| Элемент | Размер |
|---|---:|
| Высота шапки | 96-112 px |
| Внутренний отступ | 16-20 px |
| Фото/аватар администратора | 52-64 px |
| Название раздела | 32-38 px |
| Навигационная кнопка | 40-44 px высота |
| Служебная кнопка назад/выход | 40-44 px высота |

### Стиль шапки

- фон: `--admin-top`;
- нижняя граница: `1px solid --admin-line`;
- без тяжелых теней;
- все кнопки в одной системе размеров;
- активный раздел выделять зеленой нижней линией или зеленой обводкой, а не новой формой кнопки.

### Бейдж администратора

Бейдж не должен быть маленькой таблеткой. Это полноценный идентификационный блок.

Состав:

- фото или инициалы в квадрате `56 x 56`, радиус `8 px`;
- ФИО;
- роль: `Системный администратор`;
- статус доступа или время сессии, если нужно;
- зеленая точка только если есть реальный активный статус.

## Навигация

### Основные разделы админки

Рекомендуемый верхний набор:

- `Сводка`;
- `Сотрудники`;
- `Справочники`;
- `Конфликты`;
- `Журнал`;
- `Выгрузки`.

Не делать каждую вторичную функцию отдельной крупной кнопкой в шапке. Вторичные действия должны жить внутри раздела.

### Стиль навигационных кнопок

```css
height: 40px;
min-width: 118px;
padding: 0 14px;
border: 1px solid var(--admin-line);
border-radius: 6px;
background: var(--admin-panel-soft);
color: var(--admin-muted);
font-size: 13px;
font-weight: 900;
text-transform: uppercase;
```

Активная:

```css
border-color: var(--admin-green);
background: color-mix(in srgb, var(--admin-green) 12%, var(--admin-panel));
color: var(--admin-green);
```

### Назад и Выйти

`Назад` и `Выйти` не должны выглядеть как основные разделы.

Решение:

- такие же размеры;
- более нейтральный фон;
- можно добавить простую иконку;
- `Выйти` можно слегка отделить вертикальным разделителем или красным hover, но не делать постоянно красной, чтобы не спорила с опасными действиями.

## Кнопки

### Размеры

| Тип | Высота | Padding | Использование |
|---|---:|---:|---|
| Основная | 40-44 px | 0 16 px | создать, сохранить, применить |
| Вторичная | 36-40 px | 0 14 px | назад, открыть, сбросить |
| Компактная | 28-32 px | 0 10 px | действия в строках |
| Иконка | 36-40 px | квадрат | календарь, поиск, закрыть |
| Опасная | 40-44 px | 0 16 px | блокировать, удалить, завершить |

### Основная кнопка

```css
border: 1px solid var(--admin-green);
border-radius: 6px;
background: var(--admin-green);
color: #ffffff;
font-weight: 900;
text-transform: uppercase;
```

### Вторичная кнопка

```css
border: 1px solid var(--admin-line);
border-radius: 6px;
background: var(--admin-panel-soft);
color: var(--admin-ink);
font-weight: 900;
text-transform: uppercase;
```

### Опасная кнопка

```css
border: 1px solid var(--admin-red);
border-radius: 6px;
background: var(--admin-red-soft);
color: var(--admin-red);
font-weight: 900;
text-transform: uppercase;
```

### Hover

Hover должен быть заметным, но без сильного прыжка:

- можно менять фон и обводку;
- можно добавлять мягкую тень;
- `transform: translateY(-1px)` максимум;
- не увеличивать размеры кнопки.

## Поля И Фильтры

Фильтры должны работать живо, без лишних кнопок `Показать`, где это возможно.

### Поле ввода

```css
height: 40px;
padding: 0 12px;
border: 1px solid var(--admin-line);
border-radius: 6px;
background: var(--admin-panel);
color: var(--admin-ink);
font-size: 14px;
font-weight: 800;
```

### Label

```css
font-size: 11px;
font-weight: 900;
color: var(--admin-muted);
text-transform: uppercase;
```

### Фильтр-строка

Фильтры размещать в одну плотную строку:

- поиск;
- роль/статус/доступ;
- дата, если нужна;
- сброс;
- экспорт, если относится к текущему списку.

Если места не хватает, переносить второстепенные фильтры во вторую строку, а не сжимать текст до нечитаемого состояния.

### Дата

Поле даты должно открывать календарь при клике по всей строке поля, не только по маленькой иконке.

## Карточки И Панели

### Основная панель

```css
border: 1px solid var(--admin-line);
border-radius: 8px;
background: var(--admin-panel);
box-shadow: var(--admin-shadow);
```

Использовать карточки для:

- метрик;
- отдельных списков;
- форм;
- правой панели действий;
- модальных окон.

Не заворачивать всю страницу в одну большую декоративную карточку.

### Метрика

Метрика должна быть компактной:

- высота `88-112 px`;
- слева цветная метка `4 px`;
- крупное число `28-34 px`;
- подпись `12-13 px`;
- вторичная строка `11-12 px`.

### Пустое состояние

Пустое состояние не должно занимать половину экрана.

```css
border: 1px dashed var(--admin-line);
border-radius: 8px;
background: color-mix(in srgb, var(--admin-panel-soft) 70%, transparent);
color: var(--admin-muted);
padding: 14px;
```

## Таблицы И Списки

Админка - это в первую очередь списки и справочники. Таблицы должны быть плотными и читаемыми.

### Табличная строка

| Параметр | Значение |
|---|---:|
| Высота строки | 44-56 px |
| Внутренний отступ | 8-12 px |
| Радиус строки | 6-8 px |
| Левая статусная полоса | 4 px |
| Зазор между строками | 6-8 px |

### Колонки

У каждой таблицы должны быть стабильные колонки. Запрещено, чтобы ширина кнопок или бейджей меняла сетку при переходе между экранами.

Рекомендуемый паттерн для списков:

```css
display: grid;
grid-template-columns:
    minmax(260px, 1.4fr)
    minmax(130px, .7fr)
    minmax(140px, .7fr)
    minmax(160px, .8fr)
    minmax(120px, auto);
gap: 12px;
align-items: center;
```

### Табличные действия

Действия в строке делать компактными:

- высота `28-32 px`;
- текст короткий;
- опасные действия не ставить рядом с основными без разделения.

## Чипы И Бейджи

Чипы используются для коротких статусов, ролей и признаков.

```css
display: inline-flex;
align-items: center;
min-height: 24px;
padding: 3px 8px;
border: 1px solid var(--admin-line);
border-radius: 5px;
font-size: 11px;
font-weight: 900;
line-height: 1;
```

### Цвета чипов

- активный: зеленая обводка + мягкая зеленая заливка;
- ожидание: желтая обводка + мягкая желтая заливка;
- заблокирован: красная обводка + мягкая красная заливка;
- архив: серая обводка + мягкая серая заливка;
- информация: голубая обводка + мягкая голубая заливка.

## Формы

Формы должны быть разделены на смысловые блоки:

- личные данные;
- роль и доступ;
- документы/фото;
- служебные настройки;
- опасная зона.

### Поля формы

Минимальная высота поля: `40 px`.

Большие textarea:

- высота `88-120 px`;
- resize только вертикальный, если нужен;
- без огромных пустых зон.

### Фото сотрудника

Фото использовать в квадратном блоке:

- `96 x 96` или `112 x 112` для карточки;
- радиус `8 px`;
- обводка `1px solid var(--admin-line)`;
- без круглых аватаров в формах.

## Модальные Окна

Модалки должны быть в стиле пульта:

- фон затемнения с blur;
- компактное окно;
- тонкая обводка;
- красная вертикальная метка для подтверждения опасных действий;
- крупный заголовок;
- кнопки внизу справа.

### Размеры

| Тип | Ширина |
|---|---:|
| Подтверждение | 420-520 px |
| Форма/причина | 520-640 px |
| Детальная карточка | 760-960 px |

Внутренний скролл в модалке допустим только для больших детальных карточек. Подтверждения и короткие формы должны помещаться без скролла на `1365 x 768`.

## DnD-Зоны

DnD в админке сохраняем, если он реально ускоряет работу.

### Стиль зоны

- высота `64-84 px`;
- обводка `1px`;
- левая статусная полоса или цветная метка;
- заголовок `14-16 px`;
- подпись `11-12 px`;
- на hover/drop: мягкая подсветка, без скачка размера.

### Запрет

Не делать DnD единственным способом выполнить действие. У пользователя должен быть обычный способ через карточку или кнопку.

## Иконки

Для админки использовать простые line-icons, если библиотека уже подключена. Если иконок нет, лучше временно использовать текст, чем рисовать неаккуратные самодельные пиктограммы.

Иконки допустимы в:

- кнопках `Назад`, `Выйти`;
- действиях календаря, поиска, печати, Excel;
- пустых состояниях;
- DnD-зонах.

Иконки не должны заменять понятные подписи в административных действиях.

## Анимации

Анимации минимальные:

- hover `120-160 ms`;
- раскрытие модалки `140-180 ms`;
- подсветка drop-zone `120 ms`;
- без постоянных декоративных движений.

Запрещено:

- прыгающие кнопки;
- изменение высоты шапки при переходе между экранами;
- анимации, которые мешают попадать мышью по строкам и кнопкам.

## Административные Экраны

### Сводка

Назначение: быстро увидеть проблемы и перейти к работе.

Состав:

- верхние метрики;
- блок `Требует внимания`;
- последние действия;
- быстрые переходы;
- открытые конфликты.

Не перегружать графиками. Админке важнее списки и быстрые действия.

### Сотрудники

Назначение: поиск, проверка, доступы, статусы.

Состав:

- фильтр-строка;
- список сотрудников;
- статусные чипы;
- правая панель действий или компактные действия в карточке;
- детальная карточка сотрудника.

Список должен быть главным элементом экрана.

### Карточка сотрудника

Назначение: редактирование конкретного сотрудника.

Состав:

- фото и ФИО;
- статус сотрудника;
- роли и доступы;
- телефон;
- документы/проживание, если есть;
- журнал изменений;
- опасная зона отдельно снизу или справа.

Не дублировать одну и ту же информацию в нескольких соседних блоках.

### Справочники

Назначение: управление служебными данными.

Состав:

- слева список справочников;
- центр список записей;
- справа редактор выбранной записи;
- поиск и статус в верхней панели.

Редактирование должно быть аккуратным и табличным, без больших пустых карточек.

### Конфликты

Назначение: разбор проблем.

Состав:

- фильтры по статусу/типу/дате;
- список конфликтов;
- приоритетная цветовая маркировка;
- действия `В работу`, `Решен`, `Отклонен`;
- обязательная причина для опасных решений.

### Журнал

Назначение: аудит действий.

Состав:

- дата/период;
- тип действия;
- пользователь;
- объект;
- таблица событий;
- экспорт.

Журнал должен быть самым табличным и спокойным экраном.

### Выгрузки

Назначение: подготовка файлов.

Состав:

- список доступных выгрузок;
- статус готовности;
- дата/период;
- кнопки `Excel`, `Печать`, `Сформировать`;
- история последних выгрузок.

## Запреты Для Редизайна

Не делать:

- большие округлые SaaS-карточки;
- декоративные градиенты без функции;
- разные шапки на разных экранах;
- разные размеры одинаковых кнопок;
- плавающие кнопки без сетки;
- кнопки `Показать` там, где фильтр может применяться живо;
- крупные пустые зоны;
- горизонтальный скролл всей страницы;
- системные браузерные alert/confirm;
- разные значения одного цвета на разных экранах;
- смешение `Manrope` и `Roboto Condensed` в новых экранах.

## Порядок Переписывания Админки

1. Вынести общую шапку админки в include.
2. Добавить общие CSS-токены `--admin-*` для дневной и ночной темы.
3. Добавить корневой контейнер `data-admin-theme` и классы `admin-day` / `admin-night`.
4. Добавить переключатель `День / Ночь` в правую часть шапки по паттерну диспетчерской кнопки темы.
5. Привести кнопки и навигацию к одному размеру.
6. Привести фильтры к живому применению без лишних кнопок.
7. Переписать список сотрудников как эталонный экран сразу в двух темах.
8. По эталону списка сотрудников привести справочники, конфликты, журнал и выгрузки.
9. Отдельно привести карточку сотрудника и форму создания.
10. Проверить все экраны на `1365 x 768` и `1920 x 1080` в дневной и ночной теме.

## Приемка Редизайна

Редизайн админки считается пригодным, если:

- все экраны используют одну шапку;
- шапка не меняет высоту при переходе;
- во всех экранах есть единый переключатель `День / Ночь` в стиле диспетчерской;
- выбранная тема сохраняется между переходами через `localStorage`;
- кнопки одинаковых типов имеют одинаковую высоту и стиль;
- статусные цвета совпадают со смыслом;
- список сотрудников, справочники, конфликты, журнал и выгрузки читаются без наложений в дневной и ночной теме;
- фильтры работают живо там, где это возможно;
- дата открывает календарь по клику на поле;
- модалки не являются системными alert/confirm;
- нет горизонтального скролла всей страницы;
- на `1365 x 768` основные рабочие зоны не разваливаются в обеих темах;
- `manage.py check` проходит;
- релевантные тесты проходят;
- Browser QA не показывает ошибок консоли, overflow и наложений текста.

## Короткое Задание Для Другого Чата

Перепиши оформление админки по документу `ПРОГРЕСС_ПРОЕКТА/40_БРЕНДБУК_АДМИНКИ_MVP.md`.

Не меняй бизнес-логику, модели, маршруты и права доступа. Работай только с шаблонами, CSS и минимальным JS для поведения интерфейса. Цель - заменить текущее разрозненное оформление админки на единый стиль: рабочая консоль с дневной и ночной темой, `Roboto Condensed`, токены `--admin-*`, единая шапка, переключатель `День / Ночь` как в диспетчерской, стабильные кнопки, плотные таблицы, живые фильтры, аккуратные модалки и единая статусная семантика.

Начни с эталона `Сотрудники` и сразу проверь его в дневной и ночной теме, затем перенеси тот же стиль на `Сводка`, `Справочники`, `Конфликты`, `Журнал`, `Выгрузки`, карточку сотрудника и форму создания.

# Стандарт справочника простоев

Дата фиксации: 04.07.2026.

## Принцип

Простои хранятся в одном общем серверном справочнике `DowntimeReason`.

Отдельные рабочие разделы в админке не являются отдельными независимыми таблицами. Это ролевые представления общего списка:

- `Общий список простоев` - все причины простоев.
- `Простои водителя самосвала` - причины, разрешенные в приложении водителя.
- `Простои машиниста экскаватора` - причины, разрешенные в приложении экскаваторщика.
- `Детальные простои механика` - технический детальный набор для механической службы.

Такой подход оставляет один источник правды, но не перегружает водителей и экскаваторщиков большим механическим справочником.

## Поля справочника

Для причины простоя используются:

- название причины;
- короткая подпись для кнопки;
- вид техники, если причина относится только к конкретному типу техники;
- связанное состояние техники `EquipmentState`;
- флаги видимости:
  - показывать водителю самосвала;
  - показывать машинисту экскаватора;
  - показывать механику;
- порядок вывода;
- активность записи;
- признак критичности.

## Правило рабочих экранов

Рабочие интерфейсы не должны придумывать локальные кнопки простоев.

Приложение Водителя берет кнопки из раздела `Простои водителя самосвала`.

Приложение Машиниста экскаватора берет кнопки из раздела `Простои машиниста экскаватора`.

Приложение Механика берет детальный список из раздела `Детальные простои механика`.

Если администратор добавил новую активную причину простоя и включил нужный флаг видимости, соответствующая кнопка должна появиться в рабочем интерфейсе этой роли без отдельной правки HTML-шаблона.

## Связь с состояниями техники

Каждая причина простоя может быть связана с `EquipmentState`.

Это нужно для единой цветовой логики:

- `waiting` - ожидание;
- `maintenance` - ТО;
- `repair` - ремонт;
- `breakdown` - поломка;
- `downtime` - прочий простой.

Если у причины простоя состояние не указано, экран может использовать fallback-логику, но новые причины нужно по возможности связывать с `EquipmentState` явно.

## Стартовый набор

Для водителя самосвала заложены причины:

- Ожидание погрузки;
- Ожидание разгрузки;
- Ожидание разгрузки ККД;
- Ожидание разгрузки СКДР;
- Ожидание фронта работ;
- Заправка;
- ТО;
- Ремонт;
- Поломка;
- БВР;
- Обед;
- Чистка кузова;
- Климатические условия;
- Прочие.

Для машиниста экскаватора заложены причины:

- Заправка;
- ТО;
- Ремонт;
- Поломка;
- БВР;
- Обед;
- Ожидание самосвалов;
- Зачистка забоя;
- Подготовка забоя;
- Перегон экскаватора;
- Климатические условия;
- Прочие.

Для механика заложены причины:

- Ремонт;
- Поломка;
- Диагностика;
- Текущий ремонт;
- Электрика;
- Гидравлика;
- Двигатель;
- Ходовая часть;
- ТО и обслуживание;
- Сварочные работы;
- Система охлаждения;
- Шиномонтажные работы;
- Программное обеспечение.

## Правило для следующих чатов

Любой чат, который делает интерфейс Водителя, Машиниста экскаватора, Механика, Диспетчера или Горного мастера, не должен создавать собственный локальный список простоев.

Нужно подключаться к `DowntimeReason.for_workplace(...)` или к серверному API/контексту, который отдает тот же ролевой набор.

## Дополнение 04.07.2026: цвет причины простоя задается справочником

Принято правило: причина простоя не является просто текстовой кнопкой. Каждая активная причина простоя должна иметь серверную семантику состояния техники.

Рабочие интерфейсы получают:

- состав кнопок простоя из `DowntimeReason.for_workplace(...)`;
- короткую подпись кнопки из `DowntimeReason.button_label`;
- цвет кнопки из `DowntimeReason.effective_color_group`;
- смысловое состояние техники из `DowntimeReason.effective_equipment_state_code`.

Если у причины явно задан `equipment_state`, используется он. Если связь не задана, сервер применяет единый fallback по названию причины и признаку критичности. Неизвестная неаварийная причина по умолчанию трактуется как `waiting` / желтый, чтобы обычные рабочие ожидания не окрашивались в аварийный красный.

Запрещено: в интерфейсах Водителя, Машиниста экскаватора, Горного мастера и Горного диспетчера придумывать локальные цвета простоев или отдельные локальные списки причин. Цвет кнопки простоя и цвет техники при активном простое должны быть следствием серверного справочника.

Технический статус на 04.07.2026: правило перенесено на `driverform.ru`, миграция `downtimes.0004_update_downtime_reason_state_semantics` применена. Серверная проверка подтвердила: штатные ожидания и перегоны имеют `waiting/yellow`, ТО и ремонт - `maintenance|repair/orange`, поломка - `breakdown/red`.

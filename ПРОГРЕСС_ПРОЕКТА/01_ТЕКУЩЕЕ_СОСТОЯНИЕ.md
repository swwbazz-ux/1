# Текущее состояние проекта

Дата обновления: 18.06.2026

## Общая идея

Проект развивается от автоматизации диспетчерской службы к единой цифровой системе управления компанией.

Долгосрочная цель: единое ядро системы, к которому подключаются разные рабочие кабинеты отделов и специалистов.

Первый практический приоритет: производственный контур, который автоматизирует получение данных из карьера и формирование отчетов.

## Что уже сформулировано

Система должна охватывать жизненный цикл сотрудника:

- трудоустройство;
- рабочий процесс;
- увольнение.

Но запускать систему нужно поэтапно. Первым этапом выбран производственный контур, потому что диспетчерская служба сейчас перегружена ручной обработкой данных и отчетов.

## Первый контур

Производственный контур включает:

- горного мастера;
- водителей самосвалов;
- экскаваторщиков или ответственных по экскаваторам;
- диспетчерскую службу;
- в дальнейшем механическую службу и руководство.

Главная задача: данные должны появляться в системе там, где они возникают, а не собираться вручную диспетчерами из множества таблиц.

## Ключевой принцип

Пользователь не должен видеть всю систему.

Каждый работает в своем кабинете:

- водитель видит ввод по своей смене и технике;
- экскаваторщик видит свой экскаватор и смену;
- горный мастер видит расстановку и отклонения;
- диспетчер видит контроль смены и отчеты;
- руководство видит сводку и показатели.

Все кабинеты работают с одним ядром данных.

## Принцип интерфейсов

Рабочие интерфейсы должны требовать от пользователей минимальное количество действий.

Не добавлять лишние подтверждения, кнопки и сложные сценарии без реальной необходимости.

Система должна по возможности сама фиксировать время, состояние и историю изменений, а пользователь должен выполнять только понятные рабочие действия.

## Предварительное техническое направление

Целевая система видится как веб-система:

- центральное ядро и база данных;
- отдельные веб-кабинеты для ролей и отделов;
- работа через браузер на телефоне, планшете или компьютере;
- серверная часть может быть реализована на Python;
- все отделы работают с одной базой, но видят только свой функционал.

Порядок мобильной разработки:

- стартуем с PWA-версии для быстрого запуска;
- позже возможно полноценное мобильное приложение для горного мастера, экскаваторщика и водителя самосвала;
- механикам может понадобиться мобильная версия для работы с ремонтами и заявками;
- админке может понадобиться мобильная версия для базового управления;
- руководству может понадобиться мобильная отчетная витрина;
- диспетчерская работает через веб-браузер на компьютере.

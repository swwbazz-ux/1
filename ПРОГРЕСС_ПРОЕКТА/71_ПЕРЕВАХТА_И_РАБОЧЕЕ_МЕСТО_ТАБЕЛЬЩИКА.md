# Перевахта и рабочее место табельщика

Дата фиксации: 18.07.2026.

Статус: функциональность подготовлена и проверена локально. Commit, push и deploy не выполнялись. Production, действующая база данных и `driverform.ru` не менялись.

## Назначение контура

Контур заменяет ручной сбор сведений о перевахте единым серверным процессом:

1. Табельщик создает сбор на конкретную целевую вахту и указывает срок ответа.
2. При открытии сбора сервер фиксирует список всех действующих сотрудников и кадровые snapshot-данные на момент запуска.
3. Сотрудник в приложении выбирает один вариант: заезд на вахту, выезд с вахты, поездка не требуется или запрос на продление.
4. Для поездки сотрудник указывает даты, полный маршрут, транспорт, способ трансфера и смену следующей вахты.
5. Для продления сотрудник указывает период, дневную или ночную смену и причину.
6. Заявка на продление поступает начальнику участка. Только одобренная заявка возвращается табельщику в очередь документального оформления.
7. Табельщик выгружает итоговую книгу Excel, получает безопасный пакет исходных данных DOCX и отмечает завершение оформления.

Общий адрес для сотрудника: `/my/rotation/`. Ссылка не содержит идентификатор чужого ответа: после входа сервер сам открывает актуальную карточку текущего сотрудника.

## Фактический охват компании

Рабочая площадка одна: **Горный участок №2 на Малмыжском месторождении, площадка ООО «Амур Минералс»**. Запрос на заявку на билеты направляется всем действующим сотрудникам компании через их персональное приложение. Подразделение, график и бригада нужны для контроля и выгрузки, но не используются для исключения сотрудников из сбора.

Порядок перевахты:

- один состав выезжает с участка 14-го числа;
- следующий состав начинает заезжать 13-го числа;
- оставшаяся часть следующего состава заезжает 14-го числа;
- точные даты и маршрут фиксируются отдельно по каждому сотруднику, поэтому общий цикл не подменяет индивидуальные сведения одной датой.

Read-only сверка production-базы 18.07.2026 подтвердила:

- 490 карточек сотрудников всего;
- 485 действующих сотрудников, у всех 485 заполнен телефон;
- данные уже разделены по официальным подразделениям, графикам и бригадам;
- основной график сменности №12 содержит четыре бригады;
- активный доступ к существующим приложениям есть у 334 действующих сотрудников, поэтому персональное приложение должно выдать доступ также оставшимся 151 сотруднику.

Состав компании не нужно загружать в модуль отдельным файлом: при открытии сбора он берется непосредственно из действующих карточек `Employee` на сервере.

## Роли и границы ответственности

### Табельщик

- создает, открывает и закрывает сборы;
- контролирует количество приглашенных, ответивших и не ответивших;
- фильтрует ответы по состоянию, варианту и смене;
- при необходимости корректирует конкретный ответ от имени рабочего процесса;
- видит решение начальника участка и статус документального оформления;
- получает Excel по всему сбору и DOCX только по одобренным продлениям;
- отмечает, что оформление одобренного продления завершено.

### Сотрудник

- заполняет только собственный ответ;
- не видит ответы других сотрудников;
- может использовать любой свой действующий доступ к системе; отдельная роль `employee_portal` предусмотрена для сотрудников без другого рабочего приложения;
- после окончания срока не может самостоятельно изменить ответ и обращается к табельщику;
- после решения начальника участка не может изменить уже рассмотренную заявку на продление.

### Начальник участка

- видит отдельную очередь заявок на продление;
- получает ФИО, период, причину и планируемую смену `День / Ночь`;
- одобряет заявку с необязательным комментарием;
- отклоняет заявку только с обязательной причиной;
- не оформляет кадровые документы и не меняет расстановку по технике.

## Серверная модель

Создано приложение Django `rotations` с четырьмя сущностями:

- `RotationCollectionCycle` — сбор на целевую `WatchPeriod`, срок ответа, состояния `Черновик / Открыт / Закрыт / Архив`, ревизия и ответственные;
- `RotationResponse` — один неизменяемый кадровый snapshot и один ответ на пару `сбор + сотрудник`;
- `WatchExtensionCase` — период продления, решение начальника участка и отдельный статус документального оформления;
- `RotationActionLog` — серверный аудит открытия, ответов, решений и завершения оформления.

Добавлены миграции:

- `rotations.0001_initial` — таблицы, индексы, ограничения и связи контура;
- `users.0015_seed_rotation_roles` — роли `timekeeper`, `site_manager` и `employee_portal`.

Все изменения состояния выполняются транзакционно с блокировкой актуальной записи. Один открытый сбор на одну целевую вахту защищен ограничением базы данных.

## Правила смены

При создании строк сервер пытается взять `День / Ночь` из действующей принятой постоянной расстановки `EquipmentAssignment`. Источник смены сохраняется отдельно:

- действующая расстановка;
- сотрудник;
- табельщик;
- источник не определен.

Сотрудник или табельщик может уточнить смену для следующей вахты. Этот контур только подготавливает сведения для будущего обновления расстановки и не меняет `CrewPlan`, `EquipmentAssignment`, технику или активную смену автоматически.

## Защита процесса продления

- период продления не может завершаться раньше даты начала;
- причина и смена для заявки обязательны;
- решение разрешено только для заявки в состоянии ожидания;
- при отказе причина начальника участка обязательна;
- документальное оформление доступно только после одобрения;
- повторное непрерывное продление блокируется, если предыдущее одобренное продление примыкает к новому периоду без межвахтового отдыха;
- каждое действие повышает ревизию сбора и записывается в аудит.

## Выгрузка Excel

Книга содержит четыре листа:

1. `Сводка` — параметры сбора, целевая вахта, ревизия и контрольные количества.
2. `Маршруты` — только ответы о заезде и выезде с датами, маршрутом, транспортом, трансфером и сменой.
3. `Продление к оформлению` — только одобренные начальником участка кандидаты, решение и статус документов.
4. `Контроль ответов` — все сотрудники, в том числе не ответившие.

В пользовательских строках нейтрализованы значения, похожие на Excel-формулы. Даты записываются как даты, а не как текст. Листы имеют автофильтр, закрепленную строку заголовка, область печати, A4 landscape, повтор заголовка и подгонку по ширине.

## Автоматизация документов

Текущий DOCX намеренно формируется как `ПАКЕТ ИСХОДНЫХ ДАННЫХ — НЕ ПРИКАЗ`. В него входят только одобренные кандидаты, кадровые snapshot-данные, срок, смена, причина, решение и ответственные.

Система пока не создает от имени работодателя приказ, письменное согласие или иной официальный кадровый документ. Для следующего этапа нужны утвержденные организацией шаблоны Word с обязательными реквизитами и подтвержденный маршрут согласования с ОУП/кадрами. После получения шаблонов данные из одобренной заявки можно подставлять автоматически без повторного ручного ввода.

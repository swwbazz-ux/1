# Перенос проекта на новый сервер

Дата старта направления: 03.07.2026.

## Цель

Перенести текущий рабочий контур учетной системы с действующего production-сервера на новый сервер без потери кода, базы данных, статических файлов, PWA-обновлений и пользовательского доступа к интерфейсам.

Главный критерий успеха: после переноса рабочие страницы системы открываются в браузере у пользователей, включая диспетчерские и мобильные PWA-экраны, а не только доступен SSH или запущен systemd-сервис.

## Текущий известный production-контур

На 02.07.2026 действующий контур описан так:

- домены: `driverform.ru`, `www.driverform.ru`, `demo.driverform.ru`;
- текущий сервер: HOSTKEY RU, IP `77.91.93.47`;
- ОС: Ubuntu 24.04;
- приложение: `/srv/accounting-mvp`;
- пользователь запуска: `deploy`;

## Что нужно перенести

1. Код Django-проекта:
   - backend-приложения;
   - templates;
   - static;
   - management-команды;
   - service worker и PWA-слой;
   - realtime-клиент.

2. База данных PostgreSQL:
   - схема;
   - справочники;
   - пользователи и роли;
   - текущие назначения, смены, рейсы, события, журналы;
   - служебные версии realtime-состояния.

3. Серверная конфигурация:
   - `.env`;
   - секреты базы;
   - systemd unit;
   - nginx site config;
   - SSL/Let's Encrypt;
   - collectstatic/staticfiles;
   - права на каталоги;
   - firewall.

4. Резервные копии:
   - архив кода перед переносом;
   - дамп PostgreSQL перед переносом;
   - копия nginx/systemd/env-конфигов;
   - проверка возможности восстановления из дампа.

## Что нельзя потерять

- текущие PWA-версии и механизм обновления установленного приложения;
- realtime/polling-контур `/realtime/state/`;
- gzip/static отдачу `/static/js/realtime-client.js`;
- работоспособность `/mining-master-sw.js` и `/dispatcher-sw.js`;
- доступ к мобильному Горному мастеру;
- доступ к Горному диспетчеру;
- доступ к админке;
- вход по существующим пользователям и кодам доступа;
- доменные редиректы HTTP -> HTTPS, если домен переносится.

## Порядок переноса

### 1. Инвентаризация до переноса

- зафиксировать IP/провайдера/ОС нового сервера;
- проверить SSH-доступ;
- проверить DNS-план: остается ли `driverform.ru` или будет новый домен;
- снять текущие версии PWA и realtime;
- снять список активных systemd/nginx-конфигов;
- сделать дамп PostgreSQL на старом сервере;
- сделать архив кода и конфигов старого сервера.

### 2. Подготовка нового сервера

- обновить Ubuntu;
- создать пользователя `deploy`;
- настроить SSH-ключи;
- включить firewall;
- установить nginx, PostgreSQL, Python 3.12 tooling, build-essential, libpq-dev, git;
- создать базу `accounting_mvp` и пользователя БД;
- подготовить `/srv/accounting-mvp`;
- создать `.env` и `db.env`.

### 3. Перенос приложения

- загрузить код;
- создать `.venv`;
- установить зависимости;
- применить миграции или восстановить дамп БД;
- выполнить `collectstatic --noinput`;
- настроить systemd `accounting-mvp.service`;
- настроить nginx;
- проверить локально на сервере через `curl`;
- проверить публично по IP или домену.

### 4. Перенос домена и HTTPS

- если сохраняется `driverform.ru`, поменять DNS A-записи на новый IP;
- дождаться обновления DNS;
- выпустить/перевыпустить Let's Encrypt;
- проверить редиректы `http -> https`;
- проверить `driverform.ru`, `www.driverform.ru`, `demo.driverform.ru`.

### 5. Приемочные проверки

Минимально обязательно проверить:

- `/` открывается;
- `/interfaces/` открывается;
- `/dispatcher/control/` или актуальный диспетчерский маршрут открывается после входа;
- `/mining-master/assignments/` открывается на mobile viewport;
- `/system-admin/employees/` не дает 502/SSL-ошибку;
- `/static/css/app.css` отдается;
- `/static/js/realtime-client.js` отдается;
- `/mining-master-sw.js` отдает актуальную версию;
- `/dispatcher-sw.js` отдает актуальную версию;
- `/realtime/state/` отвечает;
- `accounting-mvp` active;
- `nginx` active;
- `manage.py check` проходит;
- браузерная проверка не показывает console errors и horizontal overflow на мобильных рабочих экранах.

Отдельно: успешным перенос считается только после проверки, что интерфейс диспетчера открывается в браузере из нужной пользовательской сети. SSH-доступ и HTTP 200 с машины разработчика сами по себе не закрывают задачу.

## Открытые вопросы

- IP и характеристики нового сервера.
- Будет ли использоваться прежний домен `driverform.ru` или новый домен.
- Нужно ли переносить всю текущую production-базу или можно развернуть чистую демо-базу.
- Есть ли реальные персональные данные в текущем контуре, требующие отдельного режима обработки при переносе.
- Нужно ли останавливать старый сервер после переноса или держать его как резерв на время проверки.

## Первый рабочий шаг

Перед фактическим переносом нужно получить данные нового сервера:

- IP;
- SSH-пользователь или root-доступ;
- способ входа: пароль или ключ;
- домен, который должен смотреть на новый сервер;
- решение по базе: переносим дамп текущего PostgreSQL или разворачиваем чистую демо-базу.

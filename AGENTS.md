# Инструкция для Codex по этому проекту

Эта папка является рабочей папкой проекта учетной системы.

Проект — учетная система Copper Resources.

Текущая фактическая архитектура — Django MVP. Основной код находится в `СИСТЕМА_MVP/backend`.

PWA-контуры находятся внутри Django и реализованы через `templates`, `static`, `views`, service worker и manifest. Отдельный Node frontend-пакет в текущей структуре проекта не считается рабочим контуром, пока он не найден и не подтвержден фактически.

Нельзя выдумывать несуществующие папки, модули, сервисы и frontend-приложения. Перед любой задачей нужно определить фактический рабочий контур, маршрут/экран и конкретные связанные файлы.

При мелких UI-задачах нельзя анализировать весь проект и нельзя расширять задачу на соседние контуры без прямого указания пользователя.

Production, база данных, `.env`, миграции, deploy, collectstatic, nginx и systemd нельзя трогать без прямого разрешения пользователя.

## Типы задач

### A — мелкая UI-правка

Работать только с конкретным `template`, связанным `static`-файлом и при необходимости с конкретным `view`, который отдает этот экран.

Для типа A запрещено трогать модели, миграции, базу данных, production, deploy, настройки инфраструктуры и соседние рабочие контуры.

### B — логика одного рабочего контура

Сначала определить контур: водитель, машинист экскаватора, горный мастер, диспетчер, админка или отчеты. Менять только файлы этого контура и ближайшие серверные обработчики, если они действительно участвуют в задаче.

### C — backend/domain-логика

Перед изменениями найти доменные модели, сервисные правила, формы, обработчики и тесты, которые реально участвуют в процессе. Не смешивать доменную правку с визуальным редизайном.

### D — PWA/service worker/realtime

Сначала определить конкретный manifest, service worker, shell-версию и realtime-контракт. Не менять другие PWA-контуры по аналогии без отдельной проверки.

### E — инфраструктура/production

Любые изменения production, deploy, nginx, systemd, collectstatic, `.env`, базы данных и миграций требуют прямого разрешения пользователя и отдельного плана отката.

## Обязательное начало каждой новой сессии

Перед тем как отвечать на содержательные вопросы по проекту или продолжать работу, Codex должен открыть и прочитать:

1. `ПРОГРЕСС_ПРОЕКТА/00_ИНДЕКС.md`
2. `ПРОГРЕСС_ПРОЕКТА/01_ТЕКУЩЕЕ_СОСТОЯНИЕ.md`
3. При необходимости: `ПРОГРЕСС_ПРОЕКТА/02_ЖУРНАЛ_РЕШЕНИЙ.md`
4. При необходимости: `ПРОГРЕСС_ПРОЕКТА/04_БЛИЖАЙШИЕ_ЗАДАЧИ.md`

Если пользователь просит продолжить проект, планирование, архитектуру, производственный контур, отчеты, интерфейсы или базу данных, сначала восстановить контекст из этих файлов.

## Обязательное обновление прогресса

После каждого важного шага Codex должен сам обновлять файлы прогресса.

Важным шагом считается:

- принято новое архитектурное решение;
- изменился план работ;
- создан новый документ;
- уточнен состав MVP;
- описан новый модуль;
- изменились роли, отчеты, справочники или схема данных;
- появились новые вопросы без ответа;
- пользователь дал важное уточнение по процессу.

## Какие файлы обновлять

### `01_ТЕКУЩЕЕ_СОСТОЯНИЕ.md`

Обновлять, если изменилась общая картина проекта, текущий этап или главный фокус.

### `02_ЖУРНАЛ_РЕШЕНИЙ.md`

Обновлять, если принято важное решение. Записывать дату, решение и короткое объяснение.

### `04_БЛИЖАЙШИЕ_ЗАДАЧИ.md`

Обновлять, если поменялся ближайший план, добавились задачи или часть задач выполнена.

### Новые документы

Если тема становится большой, создавать отдельный документ в `ПРОГРЕСС_ПРОЕКТА`, например:

- `05_ПРОИЗВОДСТВЕННЫЙ_КОНТУР.md`
- `06_ОТЧЕТЫ.md`
- `07_СПРАВОЧНИКИ.md`
- `08_ИНТЕРФЕЙСЫ.md`
- `09_БАЗА_ДАННЫХ.md`
- `10_ВОПРОСЫ_БЕЗ_ОТВЕТА.md`

## Правило языка

Все проектные документы, названия модулей, ролей, справочников, отчетов и пользовательских сущностей писать на русском языке.

Английские названия допустимы только во внутреннем коде, если это необходимо для разработки.

## Рабочий принцип

Не держать важный смысл только в чате.

Если информация важна для продолжения проекта, она должна быть записана в файлы папки `ПРОГРЕСС_ПРОЕКТА`.

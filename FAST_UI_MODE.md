# Быстрый режим для мелких UI-правок

Этот режим нужен для задач типа A — мелкая UI-правка. Он ускоряет работу Codex в проекте Copper Resources и ограничивает анализ только фактическим рабочим контуром.

Проект является Django MVP. Основной код находится в `СИСТЕМА_MVP/backend`. PWA-контуры находятся внутри Django через `templates`, `static`, `views`, service worker и manifest.

## Когда применять

Быстрый режим применяется только для небольших правок одного экрана:

- поменять кнопки местами;
- изменить текст;
- поправить отступ;
- поменять порядок блоков;
- скрыть или показать элемент;
- поправить цвет, иконку или подпись;
- сделать небольшую правку одного экрана.

Если задача затрагивает серверную механику, бизнес-логику, модели, миграции, realtime, service worker, production или несколько рабочих контуров, это уже не быстрый режим.

## Правила работы

Для быстрых UI-задач Codex должен:

1. Сначала определить рабочий контур.
2. Использовать `PROJECT_MAP.md`.
3. Не анализировать весь проект.
4. Не читать все документы `ПРОГРЕСС_ПРОЕКТА`, если задача явно локальная и понятная.
5. Искать только в файлах конкретного контура.
6. Менять минимальное количество файлов.
7. Не трогать соседние контуры.
8. Не трогать `models`, `migrations`, `settings`.
9. Не трогать базу, production, deploy, `.env`.
10. Не запускать полный test suite.
11. Не запускать сервер без прямого разрешения.

## Жесткое ограничение поиска

Для задач типа A Codex должен начинать с карты проекта и файлов конкретного контура. Глобальный поиск по всему проекту допустим только если конкретный `template`, `static` или `view` не удается найти по `PROJECT_MAP.md` и локальным ссылкам.

В быстром режиме нельзя:

- читать все документы `ПРОГРЕСС_ПРОЕКТА`;
- анализировать соседние рабочие контуры;
- выполнять глобальный аудит Django-приложений;
- запускать полный test suite;
- запускать локальный сервер;
- делать deploy;
- делать push без прямого разрешения пользователя.

## Ограничение рабочей области

### Водитель

- `СИСТЕМА_MVP/backend/templates/users/driver_shift.html`;
- `СИСТЕМА_MVP/backend/users/views.py` — только если нужен view;
- `СИСТЕМА_MVP/backend/users/tests.py` — только если менялась логика.

### Машинист экскаватора

- `СИСТЕМА_MVP/backend/templates/trips/excavator_work.html`;
- `СИСТЕМА_MVP/backend/static/css/excavator-work-v55*.css`;
- `СИСТЕМА_MVP/backend/trips/views.py` — только если нужен view;
- `СИСТЕМА_MVP/backend/trips/tests.py` — только если менялась логика.

### Горный мастер

- `СИСТЕМА_MVP/backend/assignments/views.py`;
- связанный template, который реально используется маршрутом;
- `СИСТЕМА_MVP/backend/assignments/tests.py` — только если менялась логика.

### Диспетчер

- `СИСТЕМА_MVP/backend/templates/trips/dispatcher_control.html`;
- `СИСТЕМА_MVP/backend/trips/views.py` — только если нужен view;
- `СИСТЕМА_MVP/backend/trips/tests.py` — только если менялась логика.

## Проверки

Если менялся только template или CSS:

- показать diff;
- указать, что серверная логика не менялась.

Если менялся view водителя:

```powershell
..\.venv\Scripts\python.exe manage.py check
..\.venv\Scripts\python.exe manage.py test users
```

Если менялся view экскаваторщика или диспетчера:

```powershell
..\.venv\Scripts\python.exe manage.py check
..\.venv\Scripts\python.exe manage.py test trips
```

Если менялся `assignments`:

```powershell
..\.venv\Scripts\python.exe manage.py check
..\.venv\Scripts\python.exe manage.py test assignments
```

Полный test suite не запускать без отдельной причины.

## Что не делать в быстром режиме

Для быстрых UI-задач Codex не должен:

- создавать новые progress-документы;
- обновлять большие документы прогресса;
- делать архитектурные выводы;
- предлагать реструктуризацию;
- трогать deploy;
- делать push;
- утверждать, что правка видна на `driverform.ru`, если deploy не выполнялся.


## Минимальный отчет

В конце быстрой UI-задачи Codex должен кратко указать:

- какие файлы изменены;
- менялась ли серверная логика;
- какая проверка достаточна или почему проверки не запускались;
- результат только локальный или опубликован;
- был ли commit, push и deploy.

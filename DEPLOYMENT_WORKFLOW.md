# Карта публикации и тестового запуска

Дата: 08.07.2026

Документ фиксирует границы между локальной разработкой, Git, сервером и браузерной проверкой для проекта Copper Resources. Главное правило: локально сделано не равно опубликовано на `driverform.ru`.

## Текущее Git-состояние на момент составления

- текущая ветка: `codex/pilot-scenario-validation`;
- upstream: `origin/codex/pilot-scenario-validation`;
- remote: `origin` -> `https://github.com/swwbazz-ux/1.git`;
- состояние относительно upstream: локальная ветка ahead 8, behind 0;
- последний baseline commit: `552cb2f7 wip: capture current driver and excavator PWA baseline`.

Если commit есть только локально и не выполнен `push`, remote и сервер его не увидят.

## Контуры работы

### Локальная разработка

Локальная рабочая папка содержит Django MVP. Основной код находится в `СИСТЕМА_MVP/backend`. PWA-контуры водителя, экскаваторщика, горного мастера и диспетчера находятся внутри Django через `templates`, `static`, `views`, service worker и manifest.

Локальная правка видна только в рабочем дереве и, при отдельном разрешении на запуск, в локальном браузере. Она не появляется на `driverform.ru` автоматически.

### Git и remote

Git фиксирует состояние проекта в commit. Remote хранит отправленные ветки и коммиты.

Различия:

- `git status` показывает локальные незакоммиченные изменения;
- `git log` показывает локальные commit;
- `ahead N` означает, что локальные commit еще не отправлены в upstream;
- сервер не обязан сам читать локальную ветку или локальный commit.

### Тестовый или боевой сервер

По локальной проектной документации рабочий серверный контур связан со следующими сущностями:

- домен: `driverform.ru`;
- дополнительные домены: `www.driverform.ru`, `demo.driverform.ru`;
- путь приложения на сервере: `/srv/accounting-mvp`;
- systemd-сервис: `accounting-mvp.service`;
- WSGI runner: `gunicorn`;
- reverse proxy: `nginx`;
- база: PostgreSQL `accounting_mvp`;
- пользователь приложения: `deploy`;
- окружение и секреты хранятся отдельно от кода.

Проектная память и прежние публикации показывают, что сервер может обновляться не через `git pull`, а через архив/копирование выбранного локального состояния с последующими серверными проверками. Поэтому нельзя считать, что push или локальный commit автоматически меняют production.

### Браузерная проверка

Браузерная проверка отвечает только на вопрос, какая версия реально отдается пользователю по конкретному URL.

Для PWA нужно отдельно учитывать:

- какой URL проверен;
- какой service worker отдается;
- какая shell/cache version ожидается;
- не показывает ли браузер старую версию из cache;
- соответствует ли публичный CSS/JS ожидаемой версии.

## Почему локально сделано не равно опубликовано

Изменение может быть сделано локально, но не видно на `driverform.ru`, если:

- файлы изменены, но не закоммичены;
- commit сделан, но не отправлен в remote;
- commit отправлен, но deploy на сервер не выполнялся;
- deploy выполнил не тот commit или не ту ветку;
- `collectstatic` не запускался после изменения static-файлов;
- gunicorn/systemd/nginx не перезапускались, когда это было нужно для публикации;
- service worker или браузерный cache продолжают отдавать старый PWA shell;
- проверяется не тот URL, роль, экран или домен.

Запуск локального сервера не равен публикации на тестовый или боевой сервер. Локальный сервер показывает файлы из текущей рабочей папки, а `driverform.ru` показывает то, что было отдельно доставлено и применено на сервере.

## Правило для Codex

Если пользователь просит "сделать правку", Codex должен уточнить или явно указать, где окажется результат:

- только локально в рабочем дереве;
- в локальном Git commit;
- в remote после `push`;
- на тестовом или production-сервере после deploy;
- в браузере после проверки конкретного URL.

В финальном отчете нужно явно писать, был ли commit, был ли push, был ли deploy и где пользователю проверять результат.

## Запрещено без прямого разрешения

Без отдельного прямого разрешения пользователя запрещено:

- выполнять `push`;
- выполнять deploy;
- запускать `collectstatic`;
- запускать `migrate`;
- перезапускать systemd/gunicorn/nginx;
- менять `.env`;
- менять базу данных;
- заходить по SSH на production;
- трогать production-конфиги.

## Минимальный безопасный цикл для мелкой UI-правки

1. Определить рабочий контур: водитель, экскаваторщик, горный мастер, диспетчер, админка или отчеты.
2. Определить маршрут/экран и конкретные связанные `template`, `static`, `view`.
3. Изменить только нужные файлы.
4. Запустить достаточную точечную проверку.
5. Сделать отдельный commit, если пользователь разрешил фиксацию.
6. Отдельно спросить разрешение на `push` и deploy.
7. После deploy проверить конкретный URL в браузере.
8. Для PWA проверить service worker, shell/cache version и фактическую видимость правки.

## Контрольная проверка после публикации

Для каждой опубликованной правки нужно фиксировать:

- commit hash;
- ветку;
- был ли `push`;
- был ли deploy;
- какой URL проверен;
- какая PWA shell/service worker версия ожидается;
- какая версия реально отдается публично;
- видна ли правка в браузере;
- были ли выполнены серверные проверки, если они входили в разрешенный план публикации.

## Очистка production static-файлов

Очистка старых static-файлов на production допускается только отдельной задачей и только по явному списку файлов. Минимальный порядок: аудит фактических подключений -> rollback-архив удаляемых файлов -> точечное удаление -> HTTP-проверка актуальных файлов и затронутого URL. `collectstatic`, deploy, миграции и restart сервисов не запускать, если они не входят в отдельно подтвержденный план.

## Как отличать состояния

Локально сделано:

- `git status` показывает modified/untracked или локальный commit;
- изменения видны только в рабочей папке или на локальном сервере, если он был отдельно запущен.

Попало в Git:

- изменения есть в commit;
- `git log` показывает нужный commit;
- если ветка `ahead`, commit еще не отправлен в upstream.

Попало в remote:

- выполнен `push`;
- upstream содержит нужный commit;
- локальная ветка больше не ahead по этому commit.

Попало на сервер:

- deploy был выполнен по нужному commit/архиву;
- серверные файлы и static соответствуют ожидаемой версии;
- публичный URL отдает новую версию;
- для PWA подтверждены service worker и shell/cache version.

Проверено пользователем:

- открыт нужный публичный URL;
- выбрана правильная роль/сессия;
- правка видна в браузере без старого cache;
- для PWA при необходимости обновлена оболочка или сброшен старый service worker/cache.

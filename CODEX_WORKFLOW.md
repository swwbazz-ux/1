# Шаблон задачи для Codex

Используй этот шаблон перед началом работы, чтобы ограничить зону изменений.

```text
Тип задачи:
Рабочий контур:
Маршрут/экран:
Что изменить:
Что нельзя трогать:
Какая проверка достаточна:
```

## Локально сделано ≠ опубликовано на сервере

После локальной правки Codex обязан сообщать, где находится результат: только в рабочем дереве, в локальном commit, в remote после `push` или на сервере после deploy.

Если commit не push-нут, сервер его не увидит. Если deploy не выполнен, `driverform.ru` может показывать старую версию даже при чистом локальном Git. Для PWA после публикации нужно учитывать service worker, cache и ожидаемую shell version.

## Быстрая UI-правка за один короткий цикл

Для задач типа A с формулировками "быстро", "мелкая правка", "поменять кнопку", "поправить интерфейс", "переставить блок" использовать `FAST_UI_MODE.md`.

Короткий цикл: определить контур, открыть `PROJECT_MAP.md`, найти конкретный `template`/`static` и только при необходимости `view`, внести минимальную правку, показать diff, указать достаточную проверку и явно написать, что результат локальный, если не было push/deploy.

Не анализировать весь проект, не трогать соседние контуры, не запускать полный test suite и не утверждать, что правка видна на `driverform.ru`, если deploy не выполнялся.

## Быстрый режим = короткий цикл

Для задачи типа A порядок работы должен быть коротким:

1. Определить рабочий контур.
2. Найти конкретный `template`, `static` и только при необходимости `view`.
3. Внести минимальное изменение.
4. Показать diff.
5. Выполнить минимальную достаточную проверку.
6. Явно написать, где находится результат:
   - только локально;
   - в commit;
   - push выполнен;
   - deploy выполнен;
   - где проверять.

Если deploy не выполнялся, не писать, что результат уже виден на `driverform.ru`.

## Примеры

### Горный мастер

```text
Тип задачи: A — мелкая UI-правка
Рабочий контур: Горный мастер
Маршрут/экран: /mining-master/assignments/
Что изменить: конкретный блок на экране назначений
Что нельзя трогать: models, migrations, база, production, deploy, соседние контуры, общий dispatcher UI вне нужного блока
Какая проверка достаточна: точечная проверка шаблона и ..\.venv\Scripts\python.exe manage.py test assignments при изменении view
```

### Диспетчер

```text
Тип задачи: A или B
Рабочий контур: Диспетчер
Маршрут/экран: /dispatcher/control/
Что изменить: конкретный элемент диспетчерского экрана или действие диспетчера
Что нельзя трогать: водительский экран, экран экскаваторщика, миграции, база, production, deploy
Какая проверка достаточна: ..\.venv\Scripts\python.exe manage.py test trips при изменении view/логики; для CSS/template — точечная браузерная проверка по экрану
```

### Экскаваторщик

```text
Тип задачи: A или D
Рабочий контур: Машинист экскаватора
Маршрут/экран: /excavator/work/
Что изменить: конкретный блок мобильного shell или связанный PWA-ресурс
Что нельзя трогать: водительский shell, диспетчерский экран, models, migrations, база, production, deploy
Какая проверка достаточна: ..\.venv\Scripts\python.exe manage.py test trips при изменении view/логики; для PWA — проверить конкретный manifest/service worker/shell
```

### Водитель

```text
Тип задачи: A или B
Рабочий контур: Водитель
Маршрут/экран: /driver/ или /driver/shift/
Что изменить: конкретный элемент сменного экрана или действие водителя
Что нельзя трогать: экран экскаваторщика, диспетчерский экран, миграции, база, production, deploy
Какая проверка достаточна: ..\.venv\Scripts\python.exe manage.py test users при изменении view/логики; для template/static — точечная проверка экрана
```
